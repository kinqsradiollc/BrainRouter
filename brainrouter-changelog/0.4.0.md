# 0.4.0 — Unreleased

Persona injection, federation foundations (shared memory, session
registry, cross-CLI messaging), and CLI multi-agent Phase 2 (typed
delegation, parent-context snapshots, route_task, output contracts).
Persona is the anchor item — the brain has been distilling a Core
Identity for several releases, but the CLI never injected it; the
LLM only saw who the user was by accident through generic recall.

## Added

- **Core Identity injection.** Briefings now render a `### Core
  Identity` section above recall cards, sourced from the brain's
  distilled persona. The persona hash is part of the cache-stable
  prefix fingerprint, so it costs zero tokens after the first turn
  and only re-anchors when the persona body actually changes.
- **`/persona` slash command.** `/persona` shows the active persona,
  hash, cognitive count, and anchor state. `/persona refresh` calls
  the brain's distillation pipeline on demand and clears the pinned
  anchor. `/persona on|off` toggles the workspace anchor preference
  without deleting the underlying `core_identity` row. Back-compat:
  `/persona <name>` still routes to the legacy `get_persona` for
  named persona definition files.
- **`memory_persona` + `memory_persona_refresh` MCP tools.**
  Canonical reader and on-demand distillation trigger; both return a
  16-char SHA-256 content hash for prefix fingerprinting. `userId`
  is optional and falls back to the default user. The pre-existing
  `get_persona` tool is unchanged — it's a separate Registry-backed
  reader for named persona definition files.
- **`cli.personaAnchor` config knob.** New `'on' | 'off'` setting in
  `~/.config/brainrouter/config.json` (default `'on'`) — system-wide
  default for whether briefings should pin Core Identity. Layered
  with the per-workspace `personaAnchorEnabled` preference: both must
  be on for the anchor to fire. No `BRAINROUTER_*` env var consulted,
  per the 0.3.9 env→config migration.
- **`/where` persona line.** Surfaces anchor state inline next to
  the existing `brain` line: `persona pinned · 412 chars`,
  `persona no body yet`, or `persona off · cli.personaAnchor=off`.
- **`/briefing` persona row.** Once `memory_persona` has been
  queried in a turn, it appears in the source-stats table alongside
  recall, working context, and the other sources — same chars /
  records columns.
- **`/memories list [query]`.** New subcommand that surfaces stored
  records with citation history. Each card now renders a precision
  badge `· cited N · uncited M (P%)` derived from the per-record
  `citationCount` and `neverCitedCount` fields, with a `⚠️ noisy`
  flag on records below 20% precision. Heads off surprise from
  the brain's auto-archive behavior by making the signal visible
  before it fires.

## Changed

- **Briefing source plan grew a sixth slot.** `buildDefaultSourcePlan`
  now lists seven sources (was six in 0.3.9): persona, recall,
  working context, task state, explain recall, failed attempts, and
  file history. Sources are still independently gated and degrade
  silently when an MCP tool is unavailable.
- **Memory cards show citation precision wherever counts are
  available.** The badge appears in `/memory` and `/memories list`,
  which call `memory_search` and receive the full record shape
  (including `citationCount` + `neverCitedCount`). `/recall` continues
  to use `memory_recall`, whose payload doesn't carry those counts
  yet — cards from that path render without a badge. Cards with no
  recall history (denominator 0) also render no badge to avoid a
  misleading "0%" tag on freshly extracted records. Wiring the
  counts into `memory_recall` is a follow-up; tracked as PERSONA-T6
  on the 0.4.1 list.

## Federation Stage 1 — Shared-memory foundation

- **SQLite WAL hardening.** `journal_mode=WAL` is verified at boot;
  the store logs a federation-aware warning when a host filesystem
  refuses WAL (some network shares + exotic tmpfs configurations do).
  New `getJournalMode()` accessor exposes the effective mode to
  callers and tests. Concurrent reader + writer across two store
  instances on one file is covered by a new integration test.
- **Per-client install snippets.** The `/mcp install <vendor>`
  generator gains `claude-code`, `codex`, and `gemini-cli` entries,
  each pointing at the conventional dotfile path and carrying a
  federation primer note inline. `brainrouter-docs/mcp-install.md`
  gains the federation overview + example blocks for the three new
  hosts.
- **`workspaceTag` on memories.** New optional field on
  `CognitiveRecord` (default = 16-char SHA-256 hex prefix of the
  workspace root path). `RecallFilters` gains a NULL-tolerant
  `workspaceTag` filter — legacy records (no tag) and unscoped
  queries (no filter) both surface every record, so federation
  rollout is gradual. Recall pipeline pre-fetches tags via a new
  batch helper so the FTS5 schema stays untouched.

## Federation Stage 2 — Active-session registry + cross-vendor presence

- **`active_sessions` table.** Composite PK `(session_key, user_id)`
  prevents a misbehaving client from stomping another user's row by
  reusing a key. Indexed by `(user_id, last_heartbeat_at DESC)` for
  the recall-the-active-peers query path.
- **Three MCP tools.** `session_register` (idempotent — server mints
  a `sessionKey` or honors a client-supplied one; preserves
  `started_at` on conflict), `session_heartbeat` (advances
  `last_heartbeat_at` and optionally `usage_json` — deliberately
  skips `operation_log` to protect audit volume), `session_list`
  (default scope = heartbeats within the last 2 min;
  `includeStale: true` and `includeUsage: true` are explicit opt-ins).
- **CLI auto-registration.** REPL startup attaches federation via
  the new `federationRegistration.ts` runtime module — registers
  once, heartbeats every 30 s, re-registers automatically when the
  brain returns `updated: false` (e.g. after a brain restart that ran
  the sweeper).
- **`/agents --remote` slash command.** Lists peer sessions with
  `[--watch] [--usage] [--include-stale] [--json]` flags. Discovery
  hints added to `/help` and to the local `/agents` output so users
  hit the federation path without prior knowledge.
- **Live Sessions dashboard widget.** New `LiveSessionsPanel` on the
  Overview page, polling `/api/sessions` every 10 s via the new
  `useActiveSessions` hook; tokens / USD column on by default.
- **Stale-session sweeper.** Runs every minute alongside the
  extraction sweeper, drops rows whose last heartbeat is older than
  5 min. Configurable via `BRAINROUTER_SESSION_SWEEP_*`; opt-out via
  `BRAINROUTER_DISABLE_SESSION_SWEEPER=true`.
- **Per-session telemetry.** `usage_json` carries the same shape the
  CLI surfaces in `/tokens` (prompt + completion + cached tokens,
  total USD, cache savings). Last-write-wins on every heartbeat.

See [`brainrouter-docs/federation.md`](../brainrouter-docs/federation.md)
for the full lifecycle (active / stale / swept / recovered),
hard-kill semantics, and the privacy boundary.

## Tests / Verification

- **Persona — 24 new cases** across five files (7 brain vitest, 4 CLI
  briefing, 3 CLI `/where`, 7 CLI precision, 3 CLI integration).
- **Federation Stage 1 — 16 new cases**: 7 vitest for the recall
  filter, 5 node:test for the `workspaceTag` round-trip + migration
  re-boot, 2 node:test for WAL hardening (single store + two stores
  on one file concurrently), 2 CLI vendor-snippet cases for the new
  federation hosts.
- **Federation Stage 2 — 23 new cases**: 11 brain vitest for the
  three MCP tools (happy + error paths), 7 brain node:test for the
  store (composite-PK isolation, sweep, audit-volume guard), 5 CLI
  node:test for the federation lifecycle (startup register, 30 s
  heartbeat cadence, re-register-on-falsy-update recovery, `stop()`).
- **Full sweep:** 305/305 brain vitest + 14/14 brain node:test
  integration + 649/649 CLI node:test green on Stage 2.

## Notes

- Persona conflict detection, reranker-side precision boosts, and
  `memory_identity_history` are tracked as PERSONA-T6..T8 and ship in
  0.4.1 — they need additional brain-side wiring that conflicts with
  the 0.4.0 ship goal.
- **Federation Stage 2 `--watch` is currently a 2 s poll, not SSE.**
  The spec calls for an SSE-fed view; SSE plumbing through the HTTP
  MCP transport is bigger than the Stage 2 PR. Captured as a 0.4.1
  follow-up.
- **Federation Stage 3 (cross-CLI messaging — `session_inbox`,
  `session_send`, broadcast)** and **CLI multi-agent Phase 2 (typed
  delegation, parent-context snapshot, `route_task`, output
  contracts)** are still to land in this release. See
  [`../FULL_TASKS.MD`](../FULL_TASKS.MD) §4 for the full task list;
  this changelog will grow as those items merge.
- **`brainrouter run` (headless) does not currently register a
  federation session.** Single-shot agent turns aren't persistent
  peers. Adding register-then-unregister around each run is a 0.4.1
  follow-up.
