# 0.4.4 — Feature-complete (on `release/0.4.4`, unreleased)

Memory pipeline tightening + unified exec policy. 0.4.3 shipped the depth
primitives (source chunks, provenance, blackboard, tree, vault, code chunker,
benchmark gate) and wired every brain agent; 0.4.4 makes them **precise and
unavoidable** rather than available-but-optional. Theme: *tighten, don't
invent.* (The CLI multi-agent + parity feature bar this sits on top of shipped
across 0.4.0–0.4.2.)

Two rounds shipped on `release/0.4.4`: the original scope (MEM-15…23, POLICY-1,
CLI-17) plus the 2026-05-31 competitive follow-ups (POLICY-2, MEM-24, MEM-25,
BRAIN-P4-T5). Packages are **bumped to 0.4.4** but **not released** — no npm
publish, no `v0.4.4` tag, no merge to `main` yet (work continues). Full suite
green: 476 brain unit + 106 integration + 944 CLI.

## Shipped — so far

### Memory precision

- **Exact chunk-level provenance (MEM-15)** — capture links each cognitive
  record only to the source chunk(s) it actually derives from, attributed by
  salient-token overlap (`memory/source/attribution.ts`), replacing the 0.4.3
  batch-level linking that tied every record in a window to every chunk.
  `memory_verify` / `memory_provenance` surface the precise chunks. Deterministic,
  zero-LLM; tunable via `BRAINROUTER_PROVENANCE_FLOOR` / `_MAX_CHUNKS`.
- **Blackboard-default admission (MEM-16)** — extracted candidates are staged on
  the blackboard and reconciled (batch dedup + below-threshold reject) before any
  commit; only survivors proceed to the existing upsert/embed/contradiction/graph
  path, each stamped committed for audit. Duplicate/rejected candidates stay on
  the blackboard (`memory_blackboard_review`), out of long-term memory. Gated by
  `BRAINROUTER_BLACKBOARD_ADMISSION` (default on); fail-open.
- **Recall expansion refs inline (MEM-17)** — recall hits carry their precise
  `sourceChunkIds` (post MEM-15) and a covering `treeNodeId`, and the
  `<relevant-memories>` briefing shows a compact `↳ source: … · tree: …` hint
  per memory. The tools guide points at `memory_fetch_source_chunk` /
  `memory_tree_walk`, so a client can drill from a summary to its exact source in
  one hop instead of a second blind query.

### Code chunks + benchmark

- **Parser-backed code chunking (MEM-18)** — the regex first-cut (symbol starts
  only, no real end) is replaced by a char-level structural scan: a brace matcher
  for C-family/Rust (ignoring braces in strings/comments, ending at the closing
  brace or a brace-less item's `;`) and an indentation scan for Python. Top-level
  code between symbols becomes its own region. A `CodeStructureAdapter` seam lets
  a tree-sitter/LSP adapter slot in later; no new runtime deps.
- **Benchmark modes + deterministic recall config (MEM-19)** — the retrieval
  benchmark passes each mode's config to recall PER-CALL (`limitsOverride` /
  `selectionOverride` / `disableReranker` / `disableJudge`) instead of mutating
  `process.env` — fixing a global-state leak that flaked the harness. Modes:
  `baseline`, `lexmmr`, plus `rerank` (when a reranker is configured) and `judge`
  (when the relevance judge is enabled); unavailable augmentations are reported
  as skipped, not silently counted as baseline. Strict recall-floor gate retained.
  (Chunk/tree "AST" retrieval is a separate fixture, not scored by this
  record-self-retrieval harness.)

### Tree

- **Tree source/topic/global policy split (MEM-20)** — the tree domain policy
  (the source → topic → global seal hierarchy, the scene-leaf domain, and the
  build/seal thresholds) now lives in `memory/tree/policy.ts`, separate from the
  generic mechanics in `tree.ts`. Thresholds are env-overridable
  (`BRAINROUTER_TREE_{MIN_SCENE_RECORDS,LEAF_PER_PASS,SEAL_THRESHOLD}`); the engine
  autobuild + maintenance consume the policy instead of hardcoded constants.
  (Also fixed two pre-existing node-tests that leaked the local `.env` config.)

### Trust

- **Unified exec policy (POLICY-1)** — every local tool routes through the one
  `decideExecutionPolicy` surface at the dispatch guard (not just the shell):
  `actionKindForTool` maps each tool to its action and `resolveToolPolicy` returns
  the allow/ask/deny decision. Mutating actions (file edit / child spawn / shell)
  emit a `policy.decision` trace event and accumulate in a per-session audit
  trail; denials carry the policy's reason; an `ask` a silent child can't answer
  fails closed. Tools outside the access-mode inventory stay blocked.
- **Log/trace redaction + path-policy primitive (MEM-36)** — span attributes
  (tool args, prompts) are now scrubbed with the transcript redactor before they
  leave the process for an external tracing backend, and the last-resort crash
  handlers redact their stack output too (defensively — redaction failure falls
  back to raw so the handler can't itself throw). Adds a reusable, realpath-aware
  `pathPolicy` helper (`isPathWithinRoots` / `assertPathWithinRoots`, robust to
  `..` traversal, symlinks, and not-yet-created write targets) — consolidating
  the ad-hoc containment checks and serving as the primitive POLICY-3's
  external-directory gate builds on.
- **Policy profiles + egress allowlist + external-dir gate (POLICY-3)** — named,
  swappable policy presets (`readonly` / `workspace` / `trusted`) flip the whole
  safety posture (access mode, sandbox, external-write mode, egress) in one
  `/policy <name>`; `/policy` alone shows the current posture. A profile-controlled
  **external-directory gate** at the tool guard denies (or asks for) file writes
  that escape the workspace root (via the MEM-36 path primitive), and a per-host
  **egress allowlist** (`cli.egressAllowlist`, empty = unrestricted) confines
  `fetch_url` outbound traffic. Decision primitives (`externalDirectoryDecision`,
  `egressDecision`) are pure + tested.

### Governance

- **Governance dry-run expansion (MEM-21)** — `memory_governance_plan` gains a
  `scope` (`cognitive` | `storage` | `all`). The `storage` scope previews the
  0.4.3 depth tables (source documents/chunks, tree nodes, vault exports) with
  per-class reclaim estimates; only orphaned source chunks (not cited by a live
  memory) count as reclaimable. Read-only; the default cognitive scope is
  unchanged.

### Retention

- **Offload reclaimer (MEM-22)** — the working-offload `ResultCache` is now a
  proper reclaimer that protects active refs: a sliding TTL (a `get` extends the
  window), LRU eviction (drops the least-recently-used, never a recently-used
  ref), and a `reclaim(protect?)` retention pass that removes expired orphans,
  evicts over-cap, never touches a protected ref, and returns reclaim stats.
  Retention is configurable via `cli.offloadRetentionMs` / `cli.offloadMaxEntries`.

### Code-aware retrieval

- **`memory_find_related` (MEM-29)** — exploration-driven code recall. Seed from
  a source chunk id or a `file`+`line` and get the nearest code-chunk neighbours
  by symbol/identifier overlap, ranked and language-scoped, the seed excluded.
  Unlike `memory_fetch_source_chunk` (±N positional neighbours inside one
  document), this crosses files to surface callers / callees / similar
  definitions. Backed by a new `source_chunks_fts` lexical index over chunk
  content + symbol (kept in sync by triggers; one-time backfill on upgrade), so
  it covers every chunk — not just the ones already distilled into a memory. The
  ranking lives in a pure, unit-tested `code-retrieval` module that the
  code-aware reranker grows into.
- **Path-prior penalties + per-file saturation (MEM-27)** — `find_related`
  ranking now down-ranks chunks you rarely want as a *definition*: vendored
  code, build output / generated files, `.d.ts` stubs, tests, and (content-aware)
  re-export barrels — so a real source hit outranks its own test or its stub
  even when the test matched more strongly. A per-file saturation cap
  (`maxPerFile`, default 2) stops one large file from crowding out the rest of
  the top-k. Each penalty is surfaced in the hit's `reason` for transparency.
- **Code-aware reranking signals (MEM-26)** — `find_related` now reorders by code
  structure, not just lexical overlap: a **definition-boost** lifts a chunk that
  *defines* an identifier the seed references (a callee / base class) above one
  that merely mentions it; a **symbol-match** boost rewards shared symbol stems
  (so `parse config` → `parseConfig` via the same splitter the query uses); and a
  **file-coherence** boost favours other chunks of the seed's own file/directory.
  Boosts fold in before the path penalty so a penalty still dominates (a test
  that defines the symbol stays below real source), and every signal is named in
  the hit's `reason`.
- **Code symbol graph — Phase 1 (MEM-28)** — at ingest, BrainRouter now records
  intra-file call/reference edges (`from_chunk` references a symbol *defined* by
  `to_chunk` in the same file) in a `code_symbol_edges` table. `find_related`
  leads its results with the seed's direct **callees** and **callers** (tagged
  `graph:callee` / `graph:caller`) — authoritative structural neighbours that
  surface even when the seed has no lexical overlap — then fills with the
  lexical hits. Edge extraction is deterministic + conservative (exact symbol
  match, ambiguous symbols dropped, self-edges excluded); `includeEdges:false`
  opts out.
- **Cross-file import edges (MEM-28b)** — `find_related` now also resolves the
  seed FILE's *relative* imports (`import … from './x'` / `require` / Python
  `from x import`) to indexed documents and surfaces a lead chunk from each as a
  `graph:import` hit — so a callee living in an imported file shows up even with
  zero lexical overlap. Resolved lazily at query time (order-independent across
  documents), bounded, and suppressed by `includeEdges:false`. (Cross-file
  dynamic dispatch remains out of scope.)
- **Incremental code-index freshness (MEM-30)** — `memory_reindex_source` keeps
  the code index current: pass a file path + its content, and BrainRouter hashes
  it — a no-op when unchanged, but on drift the prior document is marked **stale**
  (kept for provenance, excluded from `find_related`) and the fresh content is
  re-chunked. Reverting to an exact prior version revives that document instead
  of duplicating. Cheap to call on every read/edit; gate with a size/mtime stat
  to skip shipping unchanged bytes.

### Memory that learns

- **Lessons with confidence reinforcement (MEM-32)** — a new durable `lesson`
  memory type + `memory_record_lesson` tool capture reusable insights ("always
  run the migration before seeding"). Lessons are dedup-fingerprinted on
  normalized text, so recording the same lesson again **reinforces** it —
  confidence rises asymptotically toward 1 and a corroboration counter
  increments — instead of duplicating. Lessons don't decay (`halfLifeDays:
  null`), are evidence-favored, and flow through normal recall/briefings, so a
  repeatedly-confirmed lesson rises to the top. Scoped per user.
- **Skill extraction from successful sessions (MEM-33)** — `memory_extract_skill`
  distills a reusable SOP from a session summary via the synthesis LLM, with a
  `<no-skill/>` gate that stores nothing for exploratory/trivial runs. A real
  skill is saved as a durable `lesson` tagged `kind:"skill"`, so it reinforces on
  re-extraction (reusing MEM-32) and surfaces in recall. LLM failure is
  best-effort (stores nothing, never throws); the prompt/parse gate is pure +
  tested. Auto-triggering after a successful multi-step turn is a Phase-2 wiring.

### Graph analytics

- **Graph analytics lenses (DASH-1)** — `memory_graph_analytics` exposes four
  read-only lenses over the cognitive graph: **PageRank** centrality (the most
  load-bearing entities), **broker/bridge** detection (articulation points whose
  removal fragments the graph), a **namespace overview** (entity counts by type),
  and a **shortest connection path** between two entities ("how is A related to
  B"). The algorithms live in a pure, unit-tested `graph-analytics` module
  (iterative PageRank, iterative-Tarjan articulation points, BFS path); the
  dashboard *visualization* tiles consuming this data are a Phase-2 follow-up.

### Multi-agent resilience

- **Child-failure main-loop hang fixed (ORCH-FIX)** — a spawned child agent
  failing could make the CLI stop responding: `unhandledRejection` ran
  `process.exit(1)`, and a child promise could reject unhandled. Now child
  failures are fully isolated — child promises carry a `.catch` backstop with a
  defensive failure-bookkeeping block, `wait_agents` uses `Promise.allSettled`
  (a failed child becomes a per-child result, never rejects the batch),
  `wait_agent` returns a value for a missing record instead of throwing, and an
  unhandled rejection now logs loudly but keeps the interactive session alive.
  (Patterns from a grok-cli + codex multi-agent study.)
- **Resilience audit follow-up (ORCH-FIX class)** — a sweep for the same failure
  shape (detached promises, `Promise.all` batch rejection, `process.exit` on
  background paths, blind awaits) across the CLI found and fixed two more:
  - the **worker-thread** path (`spawnWorkerThread`) was the exact analog of the
    spawn bug — its fire-and-forget promise was stored in `runningWorkers`
    without a `.catch` backstop and its failure-bookkeeping did disk I/O that
    could itself throw. Both isolated now, so a failing background worker can
    never surface as an unhandled rejection.
  - the federation inbox **replay** path (`setOnInboxText`) invoked the raw
    handler, bypassing the `dispatch` try/catch the poller uses; it now routes
    through `dispatch`, so a future async handler rejecting can't leak.
  The rest of the audit came back clean — `callMcpTool` already converts every
  throw to `{isError}` (so briefing's `Promise.all`/`callSafe` can't reject),
  `mcpPool` already uses `allSettled`, the OTEL export `fetch` already has a
  `.catch`, and `runChatTurn` already catches + surfaces errors and resets to
  idle (so every `void runChatTurn(...)` is safe).

- **Redaction regression fixtures (MEM-23)** — credential-shaped fixtures pushed
  through every write boundary assert no secret reaches the stored value:
  transcript (sensory), source chunks, and the vault export render (this PR) plus
  blackboard staging + working-offload preview (already covered). Guards against
  a future change dropping a redact call.

### CLI

- **`/reload-skills` (CLI-17)** — forces a fresh re-scan of the skill
  directories (filesystem + MCP `list_skills`), reporting where it looked and
  how many skills are available. Skills are read live, so a just-added
  `SKILL.md` is picked up without a restart.
- **Edit→verify loop (CLI-18)** — after a `write_file` / `edit_file` /
  `apply_patch`, an optional checker (`cli.postEditCheck`, e.g. `tsc --noEmit` or
  `ruff check {file}`) runs on the touched file and, if it FAILS, its error
  summary is folded into the edit tool's result — so the model sees the breakage
  *before its next turn* instead of discovering it on a later `run_command`.
  Opt-in (empty = off), timeout-bounded, and never throws (a broken checker is
  non-fatal). `{file}` substitutes the edited path; project-wide checkers ignore
  it. (A full incremental LSP client — live diagnostics + semantic nav, CLI-19 —
  is a Phase-2 follow-up; this shell-out delivers the high-value half now.)
- **Crash-checkpoint + offline prompt queue (CLI-21)** — the in-flight prompt is
  checkpointed to disk before each turn and cleared after, so a process that dies
  mid-turn leaves a recoverable record; a turn that fails with a *connectivity*
  error queues its prompt to an offline list. On the next launch, any unfinished
  or offline-queued prompts are surfaced ("⏮ N prompt(s) weren't completed —
  resend if still needed") and then cleared. All best-effort (never throws).
- **Update-available notice (CLI-22)** — a throttled, cached check (once per 24h,
  via `npm view`, bounded timeout) shows a one-line "↑ brainrouter X is available"
  notice at startup when you're behind. Fully fire-and-forget — never blocks
  startup, fails silent offline, falls back to a stale cache, and is disabled
  with `cli.updateCheck: false`.

### Competitive follow-ups (2026-05-31 review)

- **End-to-end policy consistency (POLICY-2)** — every tool (local,
  orchestration/spawn/delegate, worker, MCP) now routes through
  `resolveToolPolicy` at the dispatch guard, not just local tools (POLICY-1):
  child-spawns gate as `child_write` (denied in read mode), and every mutating
  decision emits the `policy.decision` audit/trace. Closes the spawn/worker/MCP
  policy-bypass gap.
- **Parser-backed code chunks (MEM-24)** — an optional TypeScript AST adapter
  (loaded dynamically via `createRequire('typescript')`, no forced dep) slots
  into the MEM-18 `CodeStructureAdapter` seam, giving exact symbol spans that
  handle regex/template literals, JSX, and decorators; falls back to the
  structural brace scanner when `typescript` isn't resolvable.
- **Benchmark publication + code-recall metrics (MEM-25)** — a code-recall
  benchmark scores how well the chunker isolates known symbols (publishable
  numbers, `runCodeChunkBenchmark`), and the retrieval harness now reports
  per-mode latency. (Tree-walk retrieval scoring remains future.)
- **Tree policy maturity (BRAIN-P4-T5)** — `topicKeyForScene` routes scenes into
  topic buckets (the `topic_curator` primitive), and `rollupGlobalTree` digests
  accumulated global roots into a periodic cross-topic global summary
  (`BRAINROUTER_TREE_GLOBAL_ROLLUP`), wired into the maintenance pass.

### Distribution
