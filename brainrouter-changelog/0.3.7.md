# 0.3.7 — Unreleased

Headline cycle: **Terminal UI redesign + in-terminal configuration**.

## Hotfix #3 — rebuild on Ink + React (proper TUI renderer)

The raw-stdout picker kept hitting compounding redraw edge cases —
frame creep on cursor moves, frames stacking on step transitions,
slash-popup mangling the prompt on every keystroke. Each fix
uncovered a new edge case. The root cause was architectural: we
were trying to render dynamic TUI on top of a `process.stdout.write`
+ `\x1b[...]` ANSI loop, alongside readline that also owns the
prompt line. Any time the terminal's cursor was somewhere other
than where our manually-tracked offset said it was, the next
redraw landed in the wrong region. The user's terminal output
made this visible: six wizard step frames stacked on top of each
other, slash popup repeating below the prompt with mangled
footer text on every keystroke.

The fix is to **stop owning the render loop**. Switched the
wizard / `/config` / `/login` to **Ink** (React for terminals,
ink@7 + react@19) — the same library Claude Code, OpenCode, and
the broader Node TUI ecosystem use. Ink:

- Owns stdout while mounted and diffs the cell grid between
  frames (no manual cursor math, no off-by-one, no stacking).
- Composes via React components — Frame / Picker / TextField are
  declarative, share state cleanly, and the wizard's step
  machine becomes a switch in `<WizardApp>`.
- Handles all the raw-mode / cursor-visibility / SIGWINCH
  resize / Ctrl+C bookkeeping.

What landed:

- `brainrouter-cli/src/cli/ink/Frame.tsx` — bordered panel chrome
  shared by every picker / prompt.
- `brainrouter-cli/src/cli/ink/Picker.tsx` — arrow-key picker
  with N options, optional Other free-text fallback, optional
  live preview rows. Uses `useCallback` ref pattern for the
  preview hook so it doesn't infinite-loop on inline callbacks
  (this trapped the React newcomer in me on first attempt).
- `brainrouter-cli/src/cli/ink/TextField.tsx` — framed free-text
  input on `ink-text-input`, with inline validation errors.
- `brainrouter-cli/src/cli/ink/WizardApp.tsx` — the wizard state
  machine as a single Ink component. Renders one step at a time;
  `useApp().exit()` on terminal state. Stable callbacks via
  `useCallback` + ref pattern so child components don't churn.
- `brainrouter-cli/src/cli/ink/runWizard.tsx` — entry point
  (`render(<WizardApp/>)` + await its onFinish), now used by
  `index.ts` first-run auto-trigger and `/init`.
- `brainrouter-cli/src/cli/ink/runPicker.tsx` — discrete-modal
  mount helpers (`runPicker`, `runTextField`) used by `/config`
  and `/login`.
- `brainrouter-cli/src/cli/commands/{config,login}.ts` — flipped
  internal imports to the Ink helpers. Picker shape kept
  back-compat (`theme`, `footer`, `eraseOnClose` props honored)
  so the callers didn't need rewriting.
- `brainrouter-cli/src/cli/repl.ts` — slash-suggest popup
  disabled (the readline-based version mangled output on every
  keystroke; a proper Ink-based slash menu is queued for the
  full Ink REPL rewrite in 0.3.8). Tab completion via readline's
  completer is still wired and works.

New dependencies: `ink@^7`, `react@^19`, `ink-text-input@^6`,
`ink-spinner@^5`, `@types/react@^19`. The bundle adds ~70
packages but the wizard / settings UX is now bulletproof in a
way the raw-stdout approach could never be.

Anthropic native API (`/v1/messages`) deferred to 0.3.8 — the
chat loop still drives `/v1/chat/completions`. Claude users get
the OpenRouter gateway path (works identically) until then.

Tests: 266/266 green (existing suite unchanged; the Ink
components are integration-tested via manual interactive smoke
since `expect`-based TTY emulation batches keystrokes in ways
Ink's input handler doesn't see as discrete events).

## Hotfix #2 — picker creep upward + step transitions stacking + /models discovery + Claude relabel

Field reports after the first hotfix surfaced three more issues:

1. **The picker frame creeps upward by one line on every cursor
   move.** Root cause: `lastFrameHeight = text.split('\n').length`
   counted M (the number of lines) but the cursor-up
   `\x1b[<n>F` needs M-1 (the number of newlines between them). For
   a 5-line frame the math went up 5 lines instead of 4 — landing
   ONE LINE ABOVE the top of the previous frame and erasing from
   there. The new frame then drew one line higher than the previous
   one. Each arrow key shifted the frame up by 1 row.
   Fix in [`picker.ts`](../brainrouter-cli/src/cli/wizard/picker.ts):
   track `lastFrameNewlines = (text.match(/\n/g) ?? []).length`
   directly; `\x1b[<lastFrameNewlines>F` lands EXACTLY at the top
   of the previous frame. Same fix applied to
   [`cliPrompt.ts`](../brainrouter-cli/src/cli/cliPrompt.ts)
   (LLM-tool path).

2. **Every new wizard step appeared BELOW the previous step.**
   Root cause: the picker's cleanup wrote `\n` to move the cursor
   past the frame, leaving the previous step's frame visible. The
   next step's picker then drew from the line below. Over the 6
   wizard steps the screen accumulated 6 stacked frames.
   Fix: `runFramedInput` now defaults to `eraseOnClose: true` —
   each picker erases its frame on close, so the next picker draws
   at the same screen position and the previous frame is replaced.
   The `eraseOnClose` opt is preserved as an opt-out for any
   future surface that wants the frame to stay as scrollback
   (none today). The wizard's Done summary, printed after the
   last step's picker closes, lands exactly where the AGENT.md
   step's frame was.

3. **`/models` endpoint not used.** Provider step (and `/config`
   provider editor) showed only the static curated short-list,
   missing actual models installed locally (LM Studio, Ollama) or
   newly added on the user's cloud account.
   Fix: new [`wizard/modelsApi.ts`](../brainrouter-cli/src/cli/wizard/modelsApi.ts)
   helper hits `GET <endpoint>/models` (5s timeout) to fetch the
   live list from any OpenAI-compatible server. Derives the URL
   from the chat endpoint by replacing `/chat/completions` with
   `/models` — works for OpenAI, OpenRouter, DeepSeek, LM Studio,
   Ollama, vLLM, gateway proxies. Default model is moved to the
   top of the live list so its "(default)" badge stays visible.
   On failure (no key, timeout, captive portal) the wizard falls
   back to the static catalog with a subtitle hint explaining
   why. New `models-api.test.ts` covers the URL derivation
   shapes.

Claude support:

4. **The provider picker now lists "Claude (via OpenRouter)"** as
   a first-class entry. The label was previously "Anthropic (via
   OpenRouter)" which users were missing because the original
   wizard crashed on the provider step before they could see it.
   Default model bumped to `anthropic/claude-sonnet-4.5`;
   curated list extended to include sonnet-4.5 + opus-4.1 + haiku-4.5.
   The label change makes the entry findable by users searching
   for "claude". **Native `/v1/messages` adapter** (Anthropic's
   own API with `tool_use` blocks + `thinking: { budget_tokens }`)
   is non-trivial — the CLI's chat loop drives `/v1/chat/completions`
   today. Tracked as a 0.3.8 work item; until then OpenRouter's
   OpenAI-compat gateway is the supported Claude path and works
   identically (same models, same accuracy, same pricing).

Tests: 5 new cases in `models-api.test.ts` (URL derivation for
OpenAI / OpenRouter / LM Studio / Ollama shapes + trailing-slash
handling + endpoints without `/chat/completions` suffix). Full
suite 266/266 green.

## Hotfix — wizard rebuild on new internal picker

The initial cut of Item 6 (commit `225f692`) hit two bugs on first
real-world TTY use:

1. **Render stacking on every cursor move.** The original wizard ran
   on `cliPrompt.askChoice`, which had two compounding issues —
   (a) an off-by-one in the redraw math (`text + '\n'` then
   `\x1b[<n>A` cursor-up landed inside the previous frame instead of
   at its top, leaving one stale line each iteration) and
   (b) the theme step's `onCursorChange` callback wrote a preview
   line directly to stdout, which the picker's `renderedLines`
   count didn't know about, so each arrow-key press stacked
   `[Theme] / Which theme...` lines below the live frame instead of
   replacing them.

2. **Provider step crashed with "received 7 options".** The
   LLM-callable `ask_user_choice` tool enforces a 2-4 option
   constraint (so the model presents tight, mutually-exclusive
   alternatives). The wizard re-used the same primitive for an
   internal 7-row provider catalog and tripped the guard rail.

The fix is to **stop sharing the LLM-tool picker with internal CLI
flows**. The LLM tool's 2-4 limit is load-bearing — widening it
would weaken the system-prompt contract the model is taught to
follow. Instead, the wizard / `/config` / `/login` get a separate
picker primitive at
[`brainrouter-cli/src/cli/wizard/picker.ts`](../brainrouter-cli/src/cli/wizard/picker.ts)
with three architectural improvements:

- **Atomic frame render.** Each picker owns its rendered region
  for its lifetime. NO call site outside the picker may write to
  stdout while it's active. Preview lines from `onCursorChange`
  are returned as a `string[]` that the picker splices into its
  own frame — exactly the contract used by
  [`openSrc/codex/codex-rs/tui/src/theme_picker.rs`](../openSrc/codex/codex-rs/tui/src/theme_picker.rs)
  (`on_selection_changed` returns state, never writes stdout).
- **N options, no cap.** No 2-4 limit; the provider step's 7-row
  catalog renders cleanly.
- **`\x1b[<n>F\x1b[J` redraw.** `F` moves cursor up + to col 1 in
  one atomic operation; combined with NOT writing a trailing
  newline after the frame, the cursor lands EXACTLY at the top of
  the previous frame for the next redraw. No off-by-one.

Other surfaces fixed in the same commit:

- **`cliPrompt.askChoice` (LLM-tool path)** also got the
  off-by-one fix and the trailing-newline removal — it had the
  same bug, just no field reports yet because the LLM never asked
  for 7 options.
- **REPL shift+tab handler** now yields to BOTH `isPickerActive()`
  (LLM tool) AND `isInternalPickerActive()` (wizard / `/config` /
  `/login`) so cycling access modes can't bleed keystrokes into a
  framed picker.
- **`/config`, `/login`, and the full wizard** rewritten on the
  new picker. Every step now passes title / subtitle / badge /
  rows / footer as **props** to one render call — no more
  `printBoxedCard` followed by a separate `askChoice` (which is
  what produced the duplicate `[Welcome]` lines in the field
  report).

Visual polish landed alongside the fix (from
[`openSrc/grok-cli/src/ui/`](../openSrc/grok-cli/src/ui/) +
[`openSrc/codex/codex-rs/tui/src/bottom_pane/`](../openSrc/codex/codex-rs/tui/src/bottom_pane/)
patterns):

- **Single framed panel per step** with theme-colored border,
  bold title, muted subtitle, optional right-aligned badge
  (`Step 1 of 6`).
- **Value column** right-aligned per row (`Theme   dark`,
  `LLM provider   openai · gpt-4o-mini · ········cd34`).
- **`›` selected glyph** (lifted from
  `openSrc/grok-cli/src/ui/components/SuggestionOverlay.tsx`)
  replaces the previous `▶` — reads cleaner against bold + gray.
- **Subtle in-frame divider** between body and preview rows so
  the live-preview area is visually distinct.
- **Footer hints** use a single dim line at the bottom of the
  frame, key labels muted.
- **`promptText`** primitive for free-text entry (API key, URL)
  shares the same frame chrome — no jarring style switch when
  the wizard pivots from picker to text input.

Tests: 12 new cases in
[`brainrouter-cli/src/tests/picker.test.ts`](../brainrouter-cli/src/tests/picker.test.ts)
covering the frame renderer (title bar + badge layout, subtitle
wrap, divider only when preview present), the row formatter
(value right-alignment, `›` selected glyph, wrapped description
indent), and the helpers (`visibleLength` strips ANSI;
`padRightVisible` preserves color while padding;
`computeValueColumn` finds the widest visible value across rows).
Full suite: 250/250 green.

## Item 6 — Terminal UI redesign + in-terminal config wizard *(this PR)*

### 6a. Picker primitive extensions

[`brainrouter-cli/src/cli/cliPrompt.ts`](../brainrouter-cli/src/cli/cliPrompt.ts).

- `askChoice` opts gain three additive fields, all optional, all
  with `undefined` defaults that preserve pre-0.3.7 behaviour:
  - **`onCursorChange(index)`** — fired in the picker phase only,
    after a genuine cursor move. Theme picker uses this to repaint
    the prompt accent before commit. Pattern lifted from
    `openSrc/codex/codex-rs/tui/src/bottom_pane/list_selection_view.rs:117`
    and applied to our reducer-based picker.
  - **`prefilledOther`** — opens the picker in the synthetic "Other"
    free-text phase with the supplied string pre-loaded. The wizard
    + `/config` flows use this to drop the user straight into "press
    ENTER to accept the env-var value." Pattern from
    `openSrc/codex/codex-rs/tui/src/onboarding/auth.rs:150`
    (`ApiKeyInputState.prepopulated_from_env`).
  - **`initialCursor`** — lets the settings home panel re-open on
    the row the user just edited without snapping back to row 0.
- `initPickerState` signature changed to take an `init` object;
  callers that didn't pass one are unaffected because the new
  parameter is fully optional.

### 6b. Onboarding wizard

[`brainrouter-cli/src/cli/wizard/`](../brainrouter-cli/src/cli/wizard/).

- **`types.ts`** — `Step` enum + `STEP_ORDER` + pure `reduceWizard`
  reducer (`advance` / `back` / `abort` / `warn` / `commit`). The
  wizard's full lifecycle is driven by this reducer so the test
  suite can replay every transition without touching a TTY.
- **`providers.ts`** — `PROVIDER_CATALOG` (OpenAI, DeepSeek,
  OpenRouter, Anthropic-via-gateway, Gemini, LM Studio, Ollama)
  with per-provider env-var hint, curated model short-list, and
  `local` flag. `detectProviderFromEnv` walks the catalog in order
  and returns the first hit (so a user with `OPENAI_API_KEY` + 
  `DEEPSEEK_API_KEY` set still gets OpenAI pre-selected — order =
  precedence). `validateApiKey` returns the `Accept{warning?}` /
  `Reject(reason)` tier; warns on unknown prefixes rather than
  blocking. `maskApiKey` keeps the last 4 chars visible.
- **`runner.ts`** — drives the state machine over `askChoice`.
  Two entry modes:
  1. **First-run** (`ownsReadline: true`) — `index.ts` chat command
     calls this BEFORE constructing the Agent / MCP client when
     `getConfigPath()` is missing or `.onboarded` marker is absent.
     The runner creates its own readline, publishes it via
     `setActiveReadline`, walks the wizard, commits, and closes the
     readline. Non-TTY context throws `NoTTYError` (mirrors the
     `askChoice` contract).
  2. **`/init`** (`ownsReadline: false`) — REPL handler reuses the
     existing readline.
- **Commit** writes `~/.config/brainrouter/config.json` (LLM +
  MCP profile) + `<workspace>/.brainrouter/cli/preferences.json`
  (theme) + optionally `AGENT.md` in the workspace, then drops
  `~/.config/brainrouter/.onboarded` so future launches skip the
  auto-trigger.

### 6c. `/config` settings home panel

[`brainrouter-cli/src/cli/commands/config.ts`](../brainrouter-cli/src/cli/commands/config.ts).

- Bare `/config` opens an arrow-key picker over the full row table
  (LLM, MCP, theme, statusline, effort, mode, review policy, quiet,
  personality, editor, "View raw config", "Quit"). Each row shows
  its current value on the right (`Theme  dark`, `LLM provider 
  openai · gpt-4o-mini · ········cd34`). Selecting a row opens its
  sub-picker; Esc returns to home; Esc on home exits cleanly.
- Verb-overloaded short-cut forms:
  - `/config <key>` — print the current value.
  - `/config <key> <value>` — set and persist.
  - `/config raw` / `/config --raw` / `/config json` — scrubbed
    JSON dump (the pre-0.3.7 `/config` behaviour, preserved under a
    name).
- `parseConfigArgs` is exported as a pure function so the argument
  routing table is unit-testable. `listKnownConfigKeys` exposes the
  set of keys that `/config <key> <value>` accepts directly.

### 6d. `/login` slash command

[`brainrouter-cli/src/cli/commands/login.ts`](../brainrouter-cli/src/cli/commands/login.ts).

- Transport picker (`Local stdio` / `Local HTTP` / `Remote HTTP`) →
  field entry (URL + optional API key for remote) → 5s reachability
  probe via a throwaway `McpClientWrapper` → save the new profile
  and flip `config.activeServer` to it.
- Probe failure offers `Save anyway` / `Try a different transport`
  / `Cancel` — same UX as the wizard's MCP step. The "try
  different" branch recurses back into the handler so the user can
  re-pick without leaving the slash command.
- Legacy `brainrouter login` subcommand untouched — same behaviour
  as before, kept for users who scripted around it.

### 6e. Dispatcher + slash registry

[`brainrouter-cli/src/cli/repl.ts`](../brainrouter-cli/src/cli/repl.ts).

- New dispatchers (`tryHandleInitCommand`, `tryHandleConfigCommand`,
  `tryHandleLoginCommand`) run BEFORE the legacy `ui.ts` handlers in
  the category walker, so `/init` + `/config` are routed to the new
  flows immediately. The legacy switch cases in `ui.ts` are deleted
  — dead-but-harmless would have made future readers wonder which
  handler wins.
- `SLASH_COMMANDS` gains `/login`.
- Help registry text updated: `/init` description now reads
  "Re-run the onboarding wizard (Theme → Provider → API key →
  Model → MCP → AGENT.md)"; `/config` becomes "Settings panel;
  `/config theme dark` to set; `/config raw` for JSON dump";
  `/login` added as "In-REPL MCP profile editor".

### 6f. First-run auto-trigger

[`brainrouter-cli/src/index.ts`](../brainrouter-cli/src/index.ts).

- The chat command checks `fs.existsSync(getConfigPath()) &&
  isOnboarded()` before constructing the Agent. When either is
  false, it `await`s `runWizard({ ownsReadline: true })`. A
  successful wizard returns control to the chat command, which then
  loads the freshly-saved config and proceeds. An aborted wizard
  exits cleanly with exit code 0.
- The "Profile not found" error path now points at `/login` (the
  new way to add a profile) rather than `brainrouter login` (the
  legacy way) — the slash is what the user is already inside.

### 6g. Tests

[`brainrouter-cli/src/tests/wizard.test.ts`](../brainrouter-cli/src/tests/wizard.test.ts)
+ [`config-command.test.ts`](../brainrouter-cli/src/tests/config-command.test.ts).

26 new cases. Coverage at a glance:

- `STEP_ORDER` / `nextStep` / `prevStep` invariants.
- `reduceWizard` for every event kind on every step, including
  no-op contracts (advance from done, back from welcome, commit on
  non-done, abort sticky).
- Provider catalog shape (required ids present; at least one local
  entry).
- `findProvider` lookup + miss.
- `detectProviderFromEnv` precedence (first catalog hit wins; both
  set → OpenAI wins; none → undefined).
- `validateApiKey` tier: cloud-empty rejected, local-empty accepted,
  short-key rejected (paste-error guard), known prefix accepted
  silently, unknown prefix accepted with advisory.
- `maskApiKey` shape (last 4 visible, "(none)" for empty,
  short-key all-mask).
- Picker `prefilledOther` opens straight into "Other" with the
  text loaded and cursor on the Other row; `initialCursor`
  positions correctly and clamps to the option range; ENTER on a
  `prefilledOther` picker finalizes with the prefilled value.
- `/config` argument parser: 6 routing cases (home / raw / 3 raw
  aliases / get / set with value-trim).
- `listKnownConfigKeys` exports the full set of settable keys.

Full suite: brainrouter 262/262 + brainrouter-cli 238/238 = 500
green. `npx tsc --noEmit` clean.

### Out of scope for this PR

- **Fullscreen renderer** (Ratatui-style). Still 0.5.0.
- **OAuth / device-code auth.** API-key only this cycle.
- **Touching the MCP server's config** (`~/.config/brainrouter/server.env`).
  The wizard reads/writes only the CLI-owned files.
- **Provider OAuth flows** — Antigravity-style browser-based login
  + SSH-aware fallback. Designed in the spec; deferred.
- **Live theme persistence to a 4th file.** The theme writes to
  the same `<workspace>/.brainrouter/cli/preferences.json` as
  every other preference; we don't introduce a new file.
