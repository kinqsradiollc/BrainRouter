# 0.3.6 — Unreleased

Smarter memory recall, friendlier dashboard, more reliable agent loop.

## Features
- **Multi-workflow concurrency.** You can now keep multiple workflows in flight in the same workspace and move between them like git branches. The storage layer already created a folder per `/feature-dev` / `/spec` / `/review` / `/implement-plan` invocation; what was missing was the runtime UX. Six surfaces now make the switcher real:

  - **Per-workflow goal binding via `resolveGoalScope`.** Goals follow a priority chain — workflow > session > legacy — adapted from agentmemory's fallback-provider pattern. When a workflow is bound, `goal.json` lives at `<workspace>/.brainrouter/workflows/<slug>/goal.json` next to spec.md / tasks.md / meta.json so the goal travels with the folder. When no workflow is bound, the post-Item-1 session-scoped path still applies. Every read/write entrypoint (`readGoal`, `setGoal`, `clearGoal`, `patchGoal`) routes through the single `resolveGoalScope` decision point. The Item 1 strict-session-scope invariant survives intact — when no workflow is bound, sessions still don't see each other's goals.

  - **`/workflow switch <slug>`.** Refocus on an existing workflow without creating a new one. Validates the slug, migrates any session-scoped goal into the target folder when relevant, flips the per-user current pointer, refreshes the system prompt, and prints `Switched to workflow <slug> — goal: <status>, iteration N of <cap>`. Refuses with `WorkflowConflictError` when BOTH the source AND the target are active workflows with active goals — the user must `/goal pause` or `/goal complete` one explicitly before flipping, so two independent threads of work don't merge silently.

  - **First-switch migration with conflict resolution.** When the user has been working in session scope and runs `/workflow switch <slug>`, their session goal migrates into the target workflow's folder. Idempotent — running it twice is a no-op. Conflict-safe — if the target already has a non-complete goal, the helper surfaces `conflict: 'target-has-active-goal'` and the slash handler prompts via `askYesNo` "Import session's goal into <slug>?" with `keep-target` as the default. The losing side is ALWAYS archived under `<cliStateDir>/.brainrouter.migrated/` so nothing is silently lost; the archive lives in CLI state (per-user, gitignored), not the workspace tree (which would pollute committable workflow folders, per Item 1's invariant).

  - **`/workflow pause` + `/workflow resume <slug>`.** Single-shot conveniences over the dispatcher. Pause halts the current workflow's goal-continuation loop and prints the exact `/workflow resume <slug>` incantation to come back. Resume is sugar for `/workflow switch <slug>` + `/goal resume` in one keystroke — re-uses the migration plumbing for safety, then fires the next iteration via `runAgentTurn` so the user doesn't have to type "proceed."

  - **`/workflows` is now a real switcher dashboard.** Each row carries the artifact-presence markers (`spec.md:✓  tasks.md:✓  walkthrough.md:·`) AND the workflow's own goal column (`goal:active 3/unlimited`, `goal:paused`, `goal:limited`, `goal:—`), so the user can scan-pick a workflow without chasing `/goal status` across candidates. Existing first/second-line column structure preserved so any script tail/grep'ing the listing isn't broken; new goal column appended to the right. The current-pointer marker is now `★` (replaces the previous `← current` text). Statusline `workflow` segment gains a halt-state suffix — `wf:auth-overhaul (paused)`, `wf:auth-overhaul (blocked)`, `wf:auth-overhaul (limited)` — when the bound workflow's goal isn't active, so the prompt line scans the halt state without needing the `goal` segment too.

  - **Clobber prompt on `/feature-dev` / `/spec` / `/review`.** Spawning a fresh workflow when one's already current AND that current has an active goal now prompts before flipping the pointer — same UX shape as `GoalConflictError`. `--force` on the slash command skips the prompt (for scripting and `spawn_agent` silent-child use). The detection helper `detectCreateWorkflowConflict` lives in workflowArtifacts.ts and reads `goal.json` directly to avoid a workflowArtifacts → goalStore import cycle.

  Out of scope for 0.3.6 (deferred to 0.4.x federation): concurrent goal execution within one CLI (`isProcessing` lock stays — the spec is explicit) and cross-workflow handoff. Multi-workflow concurrency here means "have multiple workflows in flight, switch between them"; not "run them both at the same time in the same REPL." If the user needs parallel execution today, two CLI windows on the same workspace is the supported pattern. (Item 3 of the 0.3.6 cycle — final item, cycle close-out.)
- **Relevance judge** — opt-in LLM gate after the reranker that drops memories that share keywords but aren't actually relevant. Off by default; enable with `BRAINROUTER_RELEVANCE_JUDGE_ENABLED=true`. Falls back to reranker output on any failure — a flaky judge never breaks recall.
- **Dashboard Markdown, math, and diagrams** — chat, persona, and scene cards render full Markdown with LaTeX (`\[…\]` / `\(…\)`); Working Memory canvas renders Mermaid diagrams with theme awareness.
- **CLI shell redesign.** Boxed startup banner with workspace + MCP profile + goal + session + model in one block; new `/where` command collapses workspace/workflow/goal/plan/recall/children into a single screen so you don't have to chain four commands to orient yourself; statusline gains `workflow`, `goal`, `plan`, `pr` segments alongside the existing ones; `/quiet` (also `brainrouter --quiet`) hides recall tables, briefing dumps, and tool-completion previews for clean screenshots; new `brainrouter-cli/src/cli/theme.ts` consolidates chalk colors with `BRAINROUTER_THEME=dark|light|mono` (`auto` falls through to `dark`); the bare `?` keystroke now opens `/help`; a one-time idle-help hint fires after 30s of REPL silence so first-time users find `/where` and `/help`; Node's `ExperimentalWarning` and dotenv-deprecation noise no longer scrolls the banner off the top of the terminal (filter installed via a tiny CJS bin shim that runs before any ESM imports — needed because ESM hoists `import 'node:sqlite'` ahead of any top-level code in `src/index.ts`).
- **`/grill-me [--force] <task>` clarifying-questions command.** Pauses the agent for 2–5 clarifying questions before any file edit. Wraps the user's task in a CLARIFY-mode system overlay that forbids edits / shell / spawn for one turn and steers the model toward `ask_user_choice` for mutually-exclusive options. The overlay decays after the turn so subsequent plain prompts run normally. Skip-if-plan-exists guard: when the current workflow already has a `spec.md`, `/grill-me` refuses (and points at `/workflow switch <slug>`) unless `--force` is passed, so you don't re-litigate answers the user already gave. Pairs with `ask_user_choice` shipped earlier in this cycle.
- **Structured reasoning-step capture in working memory** (PR #32). The agent now offloads a structured "why" step after every non-trivial tool batch (≥3 tool calls OR any tool that returned >2KB) via `memory_working_offload` with `kind: "reasoning"`, `title: "Why: <short>"`, and a 1-paragraph decision summary. Pairs with the existing ~1,000-token payload-offload rule — that one captures the tool output, the new one captures the audit trail. The working-memory canvas renders reasoning nodes with a distinct dashed border so the why-trail is visually separable from `tool_output` and `compressed_summary` nodes on `canvas.mmd`. The next turn's briefing surfaces the working-memory section structurally: a `Recent steps:` block (the MCP's recentSteps tail, kept in injection order) plus a separate `Recent reasoning (why-trail):` block with the last 3 reasoning steps that fell off the recent tail — so a chatty tool burst can't push reasoning out of the model's view. Pre-existing free-form `WorkingStep.kind` schema; no type changes, no migration. (Item 2c of the 0.3.6 cycle.)
- **`/effort low|medium|high` reasoning-depth preference.** A session-level knob for "how hard should the model think." `medium` is today's behaviour and emits no overlay (so upgrade is a no-op for everyone who never touches the command). `low` adds a "be terse, skip ceremony, one-paragraph answers" overlay; `high` adds a "reason step-by-step, audit evidence before each tool call" overlay. When the **model name** matches a known reasoning family (`gpt-5*`, `o1`/`o3`/`o4*`, `gpt-oss-*`, `deepseek-r1`/`r2`, `deepseek-v3`/`v4*`, `qwen3*`, `magistral*`, or anything with `reasoning`/`thinking` in the name), the level is also forwarded as `reasoning_effort: 'low'|'medium'|'high'` in the chat/completions body — borrowed verbatim from OpenAI's [`ReasoningEffort`](https://platform.openai.com/docs/api-reference/chat/create#chat_create-reasoning_effort) enum so cross-vendor familiarity beats inventing our own names. Detection keys on the model name, not the endpoint, so it works uniformly across **OpenAI**, **DeepSeek**, **OpenRouter**, **LM Studio 0.3.29+** (`reasoning_effort` for `openai/gpt-oss-20b`), and **Ollama** (which exposes `low|medium|high` reasoning effort for its reasoning models on `/v1/chat/completions`). Non-reasoning models (`gpt-4o-mini`, `qwen2.5-coder`, etc.) skip the field everywhere — sending it would be a no-op at best. Anthropic-native (`claude-*` on `/v1/messages`) is intentionally not covered: it uses a different field shape (`thinking: { budget_tokens }`) and would need a separate provider adapter. Resolution: `BRAINROUTER_EFFORT` env > preference > default. New `effort` statusline segment shows `low` / `high` and hides on `medium` (mirroring the `exec` pattern from PR #35); `/where`'s Workspace block shows the level regardless, with an `(env)` tag when the env beat the preference. `/effort` and `/mode` are orthogonal — any combination is valid (fast-execute + deep-thinking is a legitimate ask). Re-resolved at every loop iteration so an in-session `/effort` flip applies to the next request without restart. (Item 2f of the 0.3.6 cycle.)
- **`/mode` + `/review-policy` consolidate the approval knobs.** Two new session-level commands replace the scattered `/yolo` / `autoApproveShell` / "ask before applying?" decisions with one mental model. `/mode planning` (default) routes every `run_command` through y/N and keeps the agent leaning toward clarify-before-act; `/mode fast` skips the y/N for **safe** commands and still gates dangerous ones (`rm -rf`, `sudo`, `dd`, `git push --force`, `mkfs`, `kubectl delete`, `curl … \| sh`, …) through the prompt. `/review-policy request` (default) keeps the agent's "ready for your approval?" prose in front of multi-file changes; `/review-policy proceed` tells it to apply and report after. `/yolo on` becomes a one-line alias that flips both axes (`/mode fast` + `/review-policy proceed`); `/yolo off` restores both defaults. `/approve` is unchanged — it's still the explicit per-workflow approval gesture. Both new prefs surface in `/where`'s Workspace block, and a new `exec` statusline segment shows `fast` (planning is hidden as the default). Existing `autoApproveShell: true` prefs are auto-migrated to `executionMode: 'fast'` + `reviewPolicy: 'proceed'` on first read of the new fields; the legacy flag stays on disk for the alias transition. Dangerous commands in **silent child** agents are now always denied regardless of mode — silent children can't answer the y/N, so this is an incidental safety improvement over the pre-0.3.6 `/yolo on` behaviour. The auto-clarify-on-planning-mode pass mentioned in the original spec is deferred to a follow-up: `/grill-me` (PR #31) already gives an explicit lever for it, and the heuristic for "when is a prompt ambiguous enough to auto-clarify?" needs observation data first.
- **`ask_user_choice` local tool with arrow-key picker.** Mid-turn the agent can pause, present 2–4 mutually exclusive options, and resume with the chosen label (or array of labels in `multiSelect` mode). UX matches Claude Code's `AskUserQuestion`: raw-mode picker with ↑/↓ to navigate, ENTER to confirm, SPACE to toggle in multi-select, q/Esc/Ctrl+C to cancel. An always-on **"Other"** row is appended to every prompt and drops to a free-text input — the user is never trapped between bad options. Pure reducer + renderer under the hood so the keystroke→state→render loop is unit-testable without faking a TTY. Pauses the parent REPL while active (`isPickerActive()` gates the existing shift+tab access-mode handler). Non-interactive runs (CI, piped, `brainrouter run`) surface a `NoTTYError` instead of defaulting to option 1 so the agent falls back to deciding itself with full context. User cancellation surfaces as `CancelledChoiceError` so "I declined to commit" is distinguishable from "I picked X". System prompt scopes the tool to "genuine ambiguity with 2–4 mutually exclusive reasonable approaches" — explicitly not a substitute for `askYesNo` confirmations or for thinking.

## Improvements
- **System-prompt trim (Item 9a).** The static portion of the system prompt fell from ~4,750 tokens to ~1,400 — a 70% cut without losing the load-bearing rules. Audit (2026-05-24) found most of the bulk was static lecturing the model could derive from tool descriptions: the "## Local Tools" / "## BrainRouter MCP Tools" name lists (~220 tokens) duplicated the structured tool descriptions the chat API ships in the `tools` field; the "Persistence on tool failure" section (~430 tokens) repeated standard recovery moves three different ways; the "Tool-call mechanics" + "Tool policy" + "Multi-Agent Orchestration" sections (~1,250 tokens) lectured the model on patterns each tool's description already documents; the "Asking the user mid-turn" + "Surfacing tool output" sections (~480 tokens) belong in the relevant tool descriptions, not in the system prompt. The new prompt keeps a single canonical statement per rule, deletes the duplicate tool-name listings, consolidates the right-vs-wrong tool-call example to one paragraph, and folds the workflow-artifact + autonomy + batching guidance into one section each. Pinned phrases (`Memory-First Workflow`, `non-negotiable`, `memory_recall`, `memory_resolve_session`, `memory_working_offload`, `Never say "I do not have information`, `update_plan`, `Why:`, `≥3`, `2KB`) all preserved verbatim so the existing prompt-assertion tests remain green. A new budget assertion in `tests/prompt.test.ts` fails if the base prompt re-grows past 1,800 tokens — guards against future drift. Also covers Item 9c's ordering rule: static identity + tool mechanics + memory section + behaviour rules FIRST, then the dynamic Runtime Context + Workspace Instructions + per-call overlays at the tail, so prompt-cache prefixes stay stable across turns in the same workspace. A second assertion in `prompt.test.ts` pins the ordering so a future refactor can't silently shuffle dynamic content to the front.
- **Memory recall gating (Item 9b).** Pre-9b every turn paid 3-10K tokens for a fresh briefing even on `thanks` / `/help` / single-word replies — `injectRecallContext` at `agent.ts:641` fired unconditionally for any non-silent agent. The new default `gated` mode skips the briefing unless one of three triggers fires: (a) turn 1 of the session (no prior briefing exists yet), (b) the turn immediately after auto-compaction (the model just lost context as collateral — replay the briefing so it isn't blind), or (c) the user message contains ≥2 entity-shaped tokens — file paths (`src/foo.ts`), `camelCase` / `snake_case` / `PascalCase` identifiers ≥5 chars, or proper nouns (mid-sentence capitalized words ≥3 chars). The entity heuristic is a cheap local regex/split pass — no LLM round-trip, no MCP call. When skipped, the prompt carries a one-line system-reminder tagged `memory-hint` so the model knows recall is available if it needs to pull it explicitly via `memory_recall` / `memory_search` / `memory_file_history`. The hint is dropped from the prompt the moment the full briefing fires (e.g. on the next entity-cue message). New env knob `BRAINROUTER_RECALL_MODE=always|gated|off`: `gated` is the default, `always` preserves the pre-9b every-turn behaviour for users who measured better outcomes with it (also handy for benchmarking baselines), `off` is the escape hatch for cost-per-token measurements. Resolution: env > default. Garbled values (`BRAINROUTER_RECALL_MODE=ludicrous`) fall through to `gated` defensively rather than crashing. New `recallHasFiredThisSession` + `recallNextTurnIsPostCompaction` flags on the Agent track gated state; they're session-boundary state, cleared by `loadHistory` / `fork` / `bootstrapSession` / `resetSessionCounters` so a `/new` / `/resume` re-pulls the briefing cleanly. `countEntityTokens` and `resolveRecallMode` are exported and unit-tested directly (8 new tests across `agent-runtime.test.ts`).
- **MCP identity + offline UX (Item 10).** When the BrainRouter MCP ("the brain") is offline, the user gets a clear signal and the model isn't lied to about which tools exist.
  - **10a — Identity tagging.** New `ServerConfig.identity?: 'brainrouter' | 'third-party'` field; auto-detection runs in priority order: explicit config field > server-profile name prefix (`brainrouter*`, case-insensitive) > URL host pattern (`*.brainrouter.cloud/.dev/.io/.com/.app`) > stdio command basename (`brainrouter` / `brainrouter-mcp`) > tool-signature fallback (first successful `listTools()` containing both `memory_recall` AND `list_skills` — the BrainRouter signature pair). Detection is cached on the wrapper after the first list, exposed via `McpClientWrapper.getIdentity()`. The `connect()` signature gained an optional `name?: string` argument so the wrapper can resolve identity from the profile name without re-threading state through every caller; back-compat preserved for the single-arg form.
  - **10b — Dynamic system prompt.** `buildSystemPrompt` now takes `connectedMcpTools?: string[]`. When the inventory lacks `memory_recall`, the full Memory-First Workflow section is replaced with a `## ⚠️ BrainRouter MCP is OFFLINE this turn` block telling the model NOT to call any BrainRouter memory or skill tools (they'll fail with "MCP server is not connected"), to surface the offline state if the user asks about past sessions, and to operate against the workspace files directly with local tools. The offline notice keeps the prompt honest — pre-10b the prompt still claimed `memory_*` tools existed, so the model wasted iterations on `-32601 Unknown tool` errors. Back-compat: when `connectedMcpTools` is undefined (older callers that don't pass the inventory), the prompt assumes the brain is online — no regression. The Agent caches `lastKnownMcpTools` after every `listTools()` and refreshes `chatHistory[0]` whenever the inventory shape changes (online ↔ offline) so the model sees the right prompt on the very next turn without restart.
  - **10c — Brain-status surfaces.** Three surfaces gain a distinct "brain" indicator separate from the generic `mcp` row, so users can tell "the BrainRouter cloud went down" from "a third-party MCP went down" (relevant once Item 11's multi-MCP work makes both possible):
    - **Banner**: new `brain` row rendered only when identity resolves to `brainrouter` — `🟢 online` when connected, `🔴 offline · cloud unreachable` when not. Third-party MCPs and `unknown` (pre-detection) skip the row, keeping the box compact.
    - **Statusline**: new `brain` segment; opt in via `/statusline mode,brain,…`. Shows `brain:🔴` / `brain:🟡` for non-default states only — online stays hidden, mirroring the `exec` + `effort` hide-when-default convention from PRs #35 + #36. (The 🟡 degraded state is reserved for Item 10d's local-only fallback, planned next.)
    - **`/where`**: Workspace block grew a `brain    <status>` line under the existing `mcp` row, shown unconditionally regardless of state (the "tell me everything" surface, vs. the statusline's hide-when-default stance).
  - **10d local-only fallback** — deferred. Reading recent entries from `<workspace>/.brainrouter/session-<key>/transcript.jsonl` when the brain is offline is a separate surface change (briefing renderer + opt-in env gate + new statusline state) that's safer to land in its own PR alongside the dashboard's offline-mode tests. Tracked for a 0.3.7 follow-up.
- **Multi-MCP foundation (Item 11, scope-limited).** The full federation story (parallel cross-MCP tool calls, MCP marketplace, capability tiers, tool-name collision policy) stays in 0.4.0 — what ships here is the user-facing plumbing on top of the existing single-active-MCP wrapper, so the surfaces are in place when Item 10's identity tagging starts mattering.
  - New `brainrouter-cli/src/cli/commands/mcp.ts` with three subcommands:
    - **`/mcp list`** — shows every configured profile (active marked with `★`) with identity tag (`brainrouter` / `third-party` / `unknown`), transport (`http` / `stdio`), online/offline/idle state, and the URL or stdio command. Identity is read from the live wrapper for the active profile and from the config field for idle profiles.
    - **`/mcp reconnect`** — closes the active wrapper and reconnects against the same profile + LLM config the CLI launched with. Re-probes tools so identity tagging refreshes (relevant if the user just installed a new MCP that exposes `memory_recall`).
    - **`/mcp tools`** — preserves the pre-Item-11 `/mcp` no-arg behaviour: namespace-grouped tool listing with counts. Moved under a subcommand so the existing muscle memory survives; bare `/mcp` is now an alias for `/mcp list` since that's the more useful default for users with multiple MCPs.
  - The pre-Item-11 `/mcp` handler in `commands/ui.ts:212-244` deleted in favour of the consolidated dispatcher in `commands/mcp.ts`. `/mcp` registered in `SLASH_COMMANDS` (tab-autocomplete already had it from a prior cycle) and `HELP_CATEGORIES.ui` with the updated description.
  - **Out of scope here**, deferred to 0.4.0 Stage 2: `McpClientPool` (parallel connections to N MCPs), tool-name collision policy (prefer BrainRouter on collision, namespaced fallback via `<server>:<tool>`), `activeServers: string[]` config field. The single `McpClientWrapper` instance in `index.ts:283/387/475` stays — adding the pool is a bigger refactor that's safer in its own release.
- **Goal-prompt deduplication (Item 9d).** Audit (2026-05-24) found goal text + budget counters injected on 3-4 surfaces every turn during a goal loop: the per-turn `goal-anchor` tagged system message, the foundational system message via `createSystemMessage`, the goal-continuation prompt, and the separate `goal-budget-steering` tagged message on final-budget turns. Plus the briefing's `memory_task_state` re-injecting the same "what we're doing now" context. Over a 15-iteration loop the same iteration counter + outcome line landed in ~30 places — roughly 8-16K tokens of pure repetition. 9d establishes single ownership per fact: the per-turn `goal-anchor` is the sole owner of goal text + budget + contract reminders, AND folds in the final-budget wrap-up directive automatically when `goalIsOnFinalBudgetTurn(goal)` flips (via `formatGoalBlock(goal, { finalBudgetTurn })`'s new options arg). The foundational system message no longer embeds the goal block (`createSystemMessage` returns just the base prompt + role overlay). The separate `goal-budget-steering` tagged message + its `buildBudgetSteeringMessage` builder are deleted; five `removeTaggedSystemMessage('goal-budget-steering')` defensive cleanup calls across `session.ts` + `workflow.ts` are deleted alongside it (dead code now that nothing sets the tag). The goal-continuation prompt no longer re-echoes `goal.text` — replaced with a one-line pointer to the anchor + a compressed 2-line drift check ("Does the next tool call advance the outcome stated in the anchor? If no, stop and either pick one that does, or call `goal_complete` / `goal_blocked`."). `buildGoalContinuationPrompt` moved out of the REPL closure into `state/goalStore.ts` alongside `formatGoalBlock` so it's unit-testable. The briefing skips `memory_task_state` when `hasActiveGoal: true` (the anchor already covers it); still fires when no goal is active so handover notes after `/goal pause` stay visible. 5 new tests across `goal.test.ts` + `memory.test.ts` + `agent-runtime.test.ts` lock in the ownership change.
- **`/goal` loops stay focused.** Goal text is re-anchored as a system message every turn; default budget is effectively unlimited (anti-spin and `/goal pause` remain the real safety nets); inline `budget: N iterations` works in the goal text itself.
- **Fan-out veto.** Phrases like "no spawn_agent" or "do this in one turn" reliably stop the agent from spawning child agents regardless of complexity score.
- **`.env` templates** reorganized into numbered sections; placeholder strings blanked so committed examples never look like real secrets. New `BRAINROUTER_RELEVANCE_JUDGE_*` tunables documented.

## Fixes
- **Goal-leakage across CLI sessions via workspace-level workflow pointer (9d-bugfix).** The first Item 3 multi-workflow release wrote `current-workflow.json` to workspace-level CLI state (`~/.brainrouter/workspaces/<encoded>/cli/current-workflow.json`) shared by every CLI process in the workspace. Any second CLI in the same workspace immediately read that pointer + the bound workflow's `goal.json` via `resolveGoalScope`'s workflow-priority branch — silently reintroducing the cross-session goal leak PR #26 originally fixed for the session-scoped storage path. Symptom: a fresh CLI session would refuse `/goal <new text>` with "A goal is already active" pointing at the workflow's old goal, AND the agent would auto-continue that goal (because the system prompt's per-turn goal-anchor read the workflow goal too). User-visible bug report: "each session now using the same goal."
  - **Fix:** workflow binding is now **per-session** with the workspace-level pointer demoted to a display-only "last used" hint.
    - New session-scope file `<session>/workflow.json` (lives under the same per-session bucket as `goal.json` / `tasks.json` / `transcript.jsonl`).
    - `setCurrentWorkflow(workspaceRoot, slug, sessionKey?)` writes BOTH the workspace hint AND the session binding when `sessionKey` is provided. Legacy callers without sessionKey still update the hint only — back-compat preserved.
    - `getCurrentWorkflow(workspaceRoot, sessionKey?)` prefers the session pointer when sessionKey is supplied. A fresh session returns `undefined` — explicitly NO fall-through to the workspace pointer (that's the load-bearing fix). Callers without sessionKey fall back to the workspace pointer for display.
    - New `getLastUsedWorkflow(workspaceRoot)` helper for the display-only "you were last on workflow X" lookup, distinct from the goal-binding path.
    - New `clearSessionWorkflow(workspaceRoot, sessionKey)` for explicit unbind (idempotent).
  - **Production callers threaded:** `createWorkflow` (now takes `sessionKey?` and passes it through to `setCurrentWorkflow`), `resolveGoalScope` (the main goal-scoping chokepoint), `detectCreateWorkflowConflict` (clobber-prompt scope), `shouldSkipGrillMe` (the `/grill-me` plan-exists guard), and every display surface (`banner.ts`, `statusline.ts`, `whereView.ts`, `commands/workflow.ts`'s `/approve` / `/workflow switch` / `/workflow pause` / `/workflow resume` / `/workflows`). The `★` marker in `/workflows` listing now reflects THIS session's binding, not the workspace pointer.
  - **Tests:** existing per-workflow goal-binding tests updated to pass `sessionKey` explicitly into `createWorkflow` and `setCurrentWorkflow` calls (under the new contract, those test scenarios are explicitly per-session). Three new regression tests in `goal.test.ts` lock the invariant: (a) session B does NOT inherit session A's workflow binding or goal even though the workspace-level "last used" hint persists, (b) legacy callers without sessionKey still get the back-compat workspace-pointer behaviour, (c) `clearSessionWorkflow` is idempotent and preserves the workspace hint. Suite total: 222 → 225.
  - **Migration story:** existing users with a populated workspace-level `current-workflow.json` will see their NEXT CLI session start UNBOUND from that workflow. They can rebind via `/workflow switch <slug>` or by running `/feature-dev` / `/spec` in the new session. The workflow's `goal.json` is untouched — switching back to it surfaces its preserved goal.
- **Backspace + shift+tab work in the REPL again.** A stray `setRawMode(false)` call at REPL startup (introduced in the original CLI commit, claiming to enable shift+tab keypress events) actually disabled raw mode that `readline.createInterface` automatically turns on for a TTY input. That broke Backspace at the `brainrouter[shell]>` prompt (readline expected the raw `0x7F` byte; in cooked mode the terminal owned it and readline's internal buffer drifted) and silently disabled shift+tab access-mode cycling (keypress events stopped firing). Removed the call; readline's default behavior is now what we wanted all along.
- **Memory paths no longer corrupted.** The extractor's JSON-escape repair was silently turning Windows paths and Unix path segments like `\bin` / `\target` / `\release` into control characters when the LLM emitted malformed JSON. Path strings now survive intact.
- **Relevance judge survives LM Studio model auto-unload.** Detects the "no models loaded" 400, waits 1.5s, and retries once.
- **Goal stops leaking across sessions.** Each CLI process now gets a fresh sessionKey (a UUID at agent startup), so two concurrent CLIs in the same workspace no longer share goal / plan / working state. Previously the MCP `memory_resolve_session` workspace-cache handed every CLI in a workspace the same UUID, which surfaced as "A goal is already active" from a prior session on every fresh launch. Memory recall is unaffected — the memory DB is userId-scoped, sessionKey is just a grouping tag. The startup banner now prints the session prefix so you can tell two CLIs apart at a glance. Belt-and-suspenders: a secondary leak path where `readGoal(workspace, sessionKey)` silently fell through to the legacy `cli/goal.json` is also closed, and any leftover legacy file is archived to `cli/.brainrouter.migrated/` on the first session-scoped goal write.
- **`/tokens` no longer mixes in prior CLI runs.** Children were pulled straight from the workspace-wide `sessions.json` with no scope, so every child agent ever spawned in the workspace got summed into "this session" — surfacing as fake "Children (31) · 959,815↑ · 68 LLM calls · 1,010,693 total" on a fresh CLI process that had run a single turn. Now filtered by `parentSessionKey === agent.sessionKey`. `/resume` and `/fork` also zero the in-process parent counters (`sessionUsage`, `memoryMetrics`) since the persisted transcript doesn't record per-call usage and the pre-switch counts belong to a different session. `/rename` left alone (same conversation, just relabelled).
- **`/tokens` memory-savings figure no longer invented.** Dropped the 5× heuristic on briefing tokens and the synthetic "Total estimated savings"; briefing tokens are now labelled as a cost (already counted in parent ↑), and offload chars are the only thing presented as a measured saving. The ratio line also stops clipping small-but-nonzero values to `0.00` — falls back to `<0.01` when below the visible threshold.
- **Intermittent CI test failure fixed** — flaky JWT-tampering assertion (~1/64 base64 collision odds), previously misdiagnosed as a Node-20 crypto incompatibility.
- **REPL no longer freezes after any spinner-based slash command or `ask_user_choice` picker.** Pre-fix symptoms: after `/working`, `/handover`, `/explain`, `/diagnostics`, `/forget`, `/persona`, `/skill-hints`, `/recall`, `/memory`, or any agent turn that called the `ask_user_choice` picker, the next prompt appeared but stdin was dead — Backspace echoed `^?`, arrow keys echoed `^[[A` literally, ENTER didn't submit. Root cause: `ora` (the spinner library) defaults `discardStdin: true`, which on each `.start()` invokes `stdin-discarder` to add a noop data listener + raw mode + `stdin.resume()`, and on each `.stop()` / `.succeed()` / `.fail()` does the inverse — `stdin.pause()` + `setRawMode(false)`. That last sequence drops readline's keypress listener as a side effect (`emitKeypressEvents`' lifecycle is tied to the data listener it manages), so even after `runAgentTurn`'s `rl.resume()` re-engages raw mode, no listener exists to translate raw bytes into keypress events. Fixed by routing every `ora()` call through a new `brainrouter-cli/src/cli/spinner.ts` factory that always passes `discardStdin: false`. The spinner still renders identically; only the stdin side effects are gone. Verified end-to-end with a standalone Node repro (keypress listeners survive the spinner lifecycle now, `readline.read` works after the spinner stops).
