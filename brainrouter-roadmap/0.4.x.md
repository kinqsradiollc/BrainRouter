# 0.4.x Roadmap — Post-federation polish

**Theme.** After federation lands in 0.4.0, these items round out the
features that depend on (or benefit from) federation primitives + the
augmented memory pipeline. Each item is scoped for a 0.4.1 / 0.4.2 /
0.4.3 release; the version stamping happens at planning time, not now.

**Lineage.** All items here came out of the peer-CLI parity audit
(Claude Code, Codex, Antigravity, [`deer-flow/`](../openSrc/deer-flow/),
[`semble/`](../openSrc/semble/)) and were sized for **M** effort. None
fit cleanly into 0.4.0's federation focus or 0.3.7's quick-wins
window.

---

## 1. Dynamic subagent definition

**Source:** Claude `Agent(name)`, Antigravity dynamic subagents.

**Today.** [`spawn_agent({role: ...})`](../brainrouter-cli/src/orchestration/tools.ts)
refuses unknown roles. We ship 5 presets: explorer, architect,
reviewer, worker, verifier. They cover ~95% of real tasks but the
long tail (UI-component-only contractor, db-migration-only specialist,
security-audit one-off) has no escape hatch.

**Change.**

- New `roleOverlay: string` optional field on `spawn_agent`. When
  present, replaces the role's system prompt with the supplied text
  (subject to a hard length cap — 4000 chars, same as `/goal`).
- Child sessions spawned with an overlay get `synthetic: true` in
  their `meta.json` so `/agents` can filter / flag them.
- A new memory record kind tracks role overlays the parent agent has
  *successfully* used, so repeated patterns surface in recall as
  reusable "skills" without the user having to author a SKILL.md.

**Size:** M (~250 LoC + tests).

## 2. Git-worktree-based session isolation

**Source:** Claude `--worktree` + `worktree.baseRef` setting.

**Today.** Two concurrent CLIs in the same workspace fight over the
working tree on every `write_file` / `apply_patch`. Item 1 (goal
leakage) and Item 3 (multi-workflow concurrency) of 0.3.6 reduce the
state-level contention; the filesystem-level contention is still
there.

**Change.**

- When `BRAINROUTER_WORKTREE_PER_SESSION=true` (or `--worktree` flag
  on launch), each new CLI session creates a fresh `git worktree add`
  at `<workspace>/.brainrouter/worktrees/<session-prefix>/`.
- All file ops in that session resolve against the worktree, not the
  primary checkout.
- A new `/workspace merge` command commits the worktree's changes
  back to the primary checkout (with diff preview + askYesNo).
- Pairs naturally with 0.3.6 Item 3 (multi-workflow concurrency) —
  one worktree per active workflow is the natural mapping.

**Size:** M–L (~500 LoC + tests + careful integration with sandbox).
**Depends on:** 0.3.6 Item 3 shipping first.

## 3. `/rewind` interactive timeline

**Source:** Claude `/rewind`.

**Today.** [`/transcript`](../brainrouter-cli/src/cli/commands/obs.ts)
shows session history; [`/fork`](../brainrouter-cli/src/cli/commands/session.ts)
duplicates a session. Neither lets the user *pick* a past turn to
fork from — `/fork` always starts from the most recent turn.

**Change.** New `/rewind` command opens an interactive picker (using
`inquirer` like the existing `brainrouter config` wizard) listing the
last 20 turns with timestamps + first-line previews. Selection
creates a new session forked from that turn's chat history (drops
everything after).

**Size:** M (~200 LoC + tests).

## 4. `/context [all|current]` — per-skill token estimates

**Source:** Claude `/context`, Codex blended token usage.

**Today.** [`/tokens`](../brainrouter-cli/src/cli/commands/obs.ts)
reports session aggregates (prompt / completion / call count). It
doesn't attribute tokens to specific skills, tools, or briefing
contributors.

**Change.** New `/context` command shows:
- Total session tokens.
- Per-skill breakdown (joining `sessions.json` per-turn usage against
  the skill-tag metadata captured on each memory record).
- Per-tool breakdown (which tools chewed up the most output).
- Per-briefing breakdown (recall vs working memory vs persona).

**Depends on** A3 (modular ranking refactor) shipping in 0.4.0 because
the per-skill attribution needs the skill-match boost to be cleanly
separable from the rest of the reranker scoring.

**Size:** M (~250 LoC + tests).

## 5. Per-agent / per-role `/effort`

**Source:** Claude agent frontmatter `effort:`.

**Today.** 0.3.6 Item 8 ships session-level `/effort low|medium|high`.
Children spawned via `spawn_agent` inherit nothing — they all get the
provider's default.

**Change.** Thread the 0.3.6 `/effort` preference into `spawn_agent`
config so children inherit (or override) the parent's level. New
optional `effort` field on the `spawn_agent` tool schema. Pairs with
the provider-aware reasoning-slot forwarding from 0.3.6 Item 8.

**Size:** S (~100 LoC; mostly plumbing).
**Depends on:** 0.3.6 Item 8 shipping first.

## 6. Code-aware chunking in recall *(from semble)*

**Source:** [`semble/src/semble/chunking/`](../openSrc/semble/src/semble/chunking/).

**Today.** When ingesting source files for recall (e.g. via
`@mentions` expansion or large `read_file` outputs that get
auto-offloaded), chunking is line-based. A function that spans 80
lines either fits in one chunk or gets split arbitrarily.

**Change.** AST-aware splits for at least TypeScript / JavaScript
(via the TS compiler API, already a dep transitively), Python (via
the `parso` shape from semble), and Go (via the `go/parser` shape).
Falls back to line-based for unsupported languages.

**Size:** M (~400 LoC + per-language tests).

## 7. Benchmark harness clone *(from semble)*

**Source:** [`semble/benchmarks/`](../openSrc/semble/benchmarks/) — `run_benchmark.py`, `speed_benchmark.py`, `token_efficiency.py`, `repos.json`, `annotations/`.

**Today.** [`brainrouter/benchmark/`](../brainrouter/benchmark/) exists
but is sparse compared to semble's published harness. We don't have
numbers to back token-efficiency claims for the recall pipeline.

**Change.** Port semble's harness shape, swapping their code-search
queries for our memory-recall queries:
- `run_benchmark.ts` — NDCG@10 / MRR over a fixed corpus of
  cognitive records + curated query → expected-record-ID pairs.
- `speed_benchmark.ts` — repo-indexing wall-clock + per-query
  latency p50 / p95.
- `token_efficiency.ts` — tokens-saved vs naive grep+read baseline
  for a representative session.
- `corpora/` — fixture data (anonymized memory dumps from real
  sessions) + `annotations/` (hand-labeled relevance for the queries).

**Size:** M (~500 LoC + corpus + CI integration).
**Why now.** Federation launch (0.4.0) is the natural moment to
publish "BrainRouter recall is N× more token-efficient than X" claims;
we need defensible numbers behind them.

## 8. Progressive skill loading *(from deer-flow)*

**Source:** [`deer-flow/backend/packages/harness/deerflow/skills/`](../openSrc/deer-flow/backend/packages/harness/deerflow/skills/) + their `skills/public/<skill>/SKILL.md` catalogue.

**Today.** Skill *bodies* are loaded on demand (via `get_skill`), but
the skill *catalog* (the list of names + 1-line descriptions) appears
in every briefing assembly. With 60+ skills installed, that's a
non-trivial slice of every prompt.

**Change.** Lazy catalog: only mention skills relevant to the active
goal text / active scene / recent recall results in the briefing. The
rest stay discoverable via `/skills` and `list_skills` (which already
work).

**Heuristic.** Reuse the recall pipeline:
- Embed the goal text + recent scene names; cosine-rank skill
  descriptions against them.
- Top-K (default 8) skills appear in the briefing catalog with full
  descriptions; the rest are summarized as "+N more skills available
  via `/skills` or `list_skills`."
- Skill memetic potential (0.3.0 feature) already pre-warms heated
  skills; this just stops *cold* skills from being mentioned.

**Pairs with** 0.3.6 Item 8 (`/effort`) for tighter token budgets:
`/effort low` could further trim the catalog cap.

**Size:** M (~300 LoC + tests + measurement against the benchmark
harness once it lands).
**Depends on:** A3 (modular ranking refactor) and item 7 (benchmark
harness) both shipping in 0.4.0 / earlier 0.4.x.

## Brain-Side Multi-Agent Roadmap

This is the separate MCP/memory-server work-stream from
[`../FEATURE_OPENHUMAN_BRAINROUTER.md`](../FEATURE_OPENHUMAN_BRAINROUTER.md).
It is **not** CLI child-agent orchestration. CLI agents retrieve memory
from the brain and may trigger MCP tools; the memory specialists below
run inside `brainrouter/src/memory/**`.

Version targets:

- **0.4.1** — Brain-agent job queue and registry. Add durable
  `memory_jobs`, `brainrouter/src/memory/agents/registry.ts`, common
  `BrainAgent` interfaces, status/retry tools, and dashboard/CLI health
  surfaces.
- **0.4.2** — Token-aware capture + source chunks + vault mirror. Add
  compaction rules for tool outputs, canonical source/chunk tables,
  idempotent ingest tools, and Markdown vault sync.
- **0.4.3** — Memory tree + blackboard commit pipeline. Add source tree
  buffers/summaries, drill-down MCP tools, candidate-memory blackboard,
  dedup/contradiction/evidence/commit chain, and governance tooling.
- **0.5.x** — Engineering sync providers and proactive situation
  reports. Start with Git, GitHub, local docs, terminal logs, browser
  artifacts, stale-task/risk reports, and MCP tool-schema drift memory.

Initial implementation directories:

| Area | Files / directories |
|---|---|
| Brain agents | `brainrouter/src/memory/agents/` |
| Job queue | `brainrouter/src/memory/store/*`, migrations, `brainrouter/src/memory/engine.ts` |
| Capture and extraction | `brainrouter/src/memory/capture.ts`, `brainrouter/src/memory/pipeline/*` |
| Recall and diagnostics | `brainrouter/src/memory/recall.ts`, `brainrouter/src/api/routes/*`, dashboard memory views |
| Shared types | `packages/types/src/memory.ts` |
| CLI observation only | `brainrouter-cli/src/cli/commands/orchestration.ts`, `brainrouter-cli/src/agent/agent.ts` for MCP-trigger/display surfaces |

### 9. Ownership contracts for parallel agents  *(spec §4, 0.4.1)*

`spawn_agents` gains a required `ownership` field (`files`, `module`,
`responsibility`, or `question`) for any entry whose `access` is
`write` or `shell`, unless `allowOverlap: true` is explicit. Read-only
fan-out emits a warning instead of rejecting. Worker children with a
path-shaped `ownership` cannot edit files outside the glob — `writeFile`
/ `editFile` / `applyPatch` return a structured tool error. `wait_agents`
result objects include ownership.

**Size:** M (~300 LoC + tests).

### 10. Tool-surface budgeting + dynamic filtering  *(spec §8, 0.4.1)*

Today every child sees the parent's full local + MCP tool catalog.
Per-agent `toolScope` (already in the definition schema from 0.3.7
Phase 1) + `disallowedTools` are now enforced. Out-of-scope tool
calls return a structured error with `suggested: [...]` alternatives.
A new `toolBudget.ts` ranks MCP tools by name-match against the
prompt + `whenToUse` and keeps top-N when the catalog exceeds the
budget (`BRAINROUTER_AGENT_MCP_TOOL_BUDGET`, default 40).

**Size:** M (~350 LoC + tests).

### 11. Supervisor gates + per-agent accounting  *(spec §14 + §15, 0.4.1)*

`/delegation-policy [auto|ask-before-spawn|ask-before-write-child|no-children]`
persisted per workspace. `handleSpawn` consults the policy before
creating the session. `ChildSessionRecord.usage` extends with
`offloadedChars` + `wallClockMs`; `/tokens` gets a "By child"
subsection; trace JSONL gains `agent_id`, `parent_agent_id`,
`ownership`, `tool_count`.

**Size:** S+S (~200 LoC total + tests).

### 12. Auto-chain (worker → reviewer → verifier)  *(spec §17, 0.4.1)*

Formalize the existing `autoReview` preference into a proper chain:
`/auto-chain [review|verify|both|off]`. Chain depth capped (default 2
follow-ups per worker). `/auto-review` becomes a thin alias.

**Size:** S (~100 LoC + tests).

### 13. Confidence-scored review fan-out  *(spec §7, 0.4.2)*

`/review-auto [--threshold 80] [--scope <glob>]` spawns N independent
reviewers (instruction-compliance, bug-risk, tests, history-context,
simplification). Reviewer `outputContract` requires
`findings[].confidence` (0-100). Parent synthesis merges duplicates
by `(file, line-range, root-cause-hash)` and filters below threshold.
Below-threshold findings stay in the child transcript for audit.

**Size:** M (~400 LoC + tests + per-reviewer prompt overlays).

### 14. Progressive result handoff  *(spec §9, 0.4.2)*

New `extract_result({ resultRef, query })` tool. Generalize the
existing `OFFLOAD_*` constants in `orchestration/tools.ts` into
`runtime/resultHandoff.ts`. Any tool result over threshold gets
`{resultRef, preview, bytes, estimatedTokens}`. Local CLI cache
first (per-session, TTL'd), durable handoff via `memory_working_offload`.
Wire `run_command`, `grep_search`, `fetch_url`, and large MCP outputs
through it.

**Brain boundary:** durable handoff lives in MCP working memory. The
CLI cache is for in-session compaction only.

**Size:** M (~400 LoC + tests).

### 15. Dedicated worker threads  *(spec §10, 0.4.2)*

`spawn_worker_thread({agentId, prompt, ownership, goal, budget})`.
Workers persist under `<workspace>/.brainrouter/cli/workers/<id>/`
(meta.json, transcript.jsonl, goal.json + plan.json, rolling
summary.md). New commands: `/workers`, `/workers attach <id>`,
`wait_worker`, `read_worker_summary`, `close_worker`. Workers cannot
spawn workers (`MAX_WORKER_DEPTH = 1`); writes are ownership-gated.

**Depends on:** federation Stage 4 (0.4.1) — workers need the same
resumption primitives as cross-CLI handoff.

**Size:** L (~700 LoC + tests).

### 16. Agent-definition packs  *(spec §13, 0.4.2)*

`.brainrouter-pack/` layout (`pack.json`, `commands/`, `agents/`,
`skills/`, `hooks/`, `mcp.json`). Slash commands: `/pack list`, `/pack
enable <name>`, `/pack disable <name>`, `/pack info <name>`. Ship
`feature-dev` and `pr-review` packs as references.

**Size:** L (~600 LoC + 2 reference packs + tests + docs).

### 17. Agent transcript debugger  *(spec §16, 0.4.2)*

`/agents tree`, `/agents why <id>`, `/agents transcript <id> --tools
--errors`, `/agents replay <id>`. All read-only by default. Builds on
the parent context snapshot from 0.4.0.

**Size:** M (~400 LoC + tests).

### 18. Delegation-aware memory capture  *(spec §18, 0.4.2)*

CLI emits structured events through MCP (`delegation_decision`,
`agent_output`, `verification_result`, `review_finding`,
`agent_route_feedback`). New `memoryEvents.ts` emitter. The brain
side decides storage shape — see
[`../FEATURE_OPENHUMAN_BRAINROUTER.md`](../FEATURE_OPENHUMAN_BRAINROUTER.md).
This closes the loop on Phase 2's `route_task` memory-aware routing
— now the routing signal exists in the brain to be queried.

**Size:** M (~300 LoC on CLI side + tests + co-ordination with brain
side).

### 19. Brain-agent awareness commands  *(spec §19, 0.4.2/0.4.3)*

`/brain agents`, `/brain jobs`, `/brain retry <jobId>`, `/brain why
<memoryId>`. All read-only from the CLI side. Gated on MCP exposing
a `brain_agent_list` / `brain_job_list` / `brain_provenance` tool
surface — that work is tracked in
[`../FEATURE_OPENHUMAN_BRAINROUTER.md`](../FEATURE_OPENHUMAN_BRAINROUTER.md);
land in 0.4.2 if MCP is ready, slip to 0.4.3 otherwise.

**Size:** M (~300 LoC + tests, contingent on brain-side API).
