# 0.5.0 Roadmap — TUI cycle + plugin marketplace

**Status.** Sketched. The two big items here (fullscreen TUI, plugin
marketplace) are each XL on their own; they group together because
they target the same audience (heavy daily users who've outgrown the
inline REPL) and because the gateway shape in [`0.4.0.md` → A5](0.4.0.md#a5--langgraph-compatible-http-gateway-hardening-from-deer-flow)
unblocks the publishing infrastructure both need.

**Lineage.** All three items emerged from the peer-CLI parity audit;
the TUI item was already flagged for 0.5.x in 0.3.6 Item 2's
"out of scope" list with [`antigravity-cli/`](../openSrc/antigravity-cli/)
as the reference design.

---

## 1. Fullscreen TUI / `/focus`

**Source:** Claude `/focus`, `/tui fullscreen`,
[`antigravity-cli/`](../openSrc/antigravity-cli/),
[`DeepSeek-TUI/`](../openSrc/DeepSeek-TUI/).

**Today.** REPL renders inline using the user's terminal scrollback.
Heavy sessions (long fan-outs, large tool previews) push earlier turns
out of view; copy-paste of a specific assistant response means hunting
in scrollback.

**Change.** Alternate-screen renderer toggled by `/focus` or `--tui`
launch flag, providing:
- **Scrollable history pane** independent of terminal scrollback (own
  PgUp/PgDn, mouse wheel, `g`/`G`).
- **Persistent status pane** showing the same content as `/where`,
  live-updating as goal / plan / agents change.
- **Mouse support** for selecting text without breaking session
  ANSI (auto-copy on select, configurable).
- **Tool-output collapsing** — long previews collapse to a one-line
  header that expands on click / Enter.
- **Inline keybinding help** — `?` overlay showing the current key
  map; `Esc` to close.

**Reference reading.** [`antigravity-cli/`](../openSrc/antigravity-cli/)
for ergonomic patterns;
[`DeepSeek-TUI/crates/tui/`](../openSrc/DeepSeek-TUI/crates/tui/) for
reasoning-block UI patterns (also flagged in 0.3.6 Item 2 follow-ups).

**Size:** L (~1500 LoC for the renderer + 500 LoC for keybindings +
tests). The cleanest path is `blessed` or `ink` rather than rolling
our own; the choice is part of the design pass.

## 2. Plugin marketplace

**Source:** Claude `/plugin`, Codex marketplace.

**Today.** [`skills/`](../skills/) folder works as a directory of
markdown skills loadable via `/skill <name>`. There's no:
- Discovery / install / update / uninstall flow.
- Versioning per skill.
- Signature verification.
- Plugin manifest schema (capabilities a skill needs, MCP servers
  it ships with, hooks it registers).
- Cross-machine distribution (today: clone-and-symlink).

**Change.** A new `/plugin` command + a registry that builds on the
existing `skills/` shape:

- **Plugin manifest** (`plugin.json` at the skill root) declaring
  name, version, capabilities (slash commands, MCP servers, hooks,
  themes), and signature.
- **Registry config** in
  `~/.config/brainrouter/registries.json` listing trusted plugin
  sources (npm packages, git URLs, HTTP indexes). Default: an
  official `kinqsradiollc/brainrouter-plugins` git repo.
- **CLI commands.** `/plugin browse` (search registry),
  `/plugin install <name>` (download + verify + symlink under
  `skills/`), `/plugin update [name]`, `/plugin uninstall <name>`,
  `/plugin validate` (schema + signature check on local).
- **Signature verification.** Plugins ship a detached signature
  (`plugin.json.sig`); the CLI checks against a key bundle in the
  registry config. Unsigned plugins install with a confirmation
  prompt.
- **Per-plugin hooks installation.** Plugin-supplied hooks land in a
  `hooks/<plugin-name>/` subfolder so they're tagged and removable
  on `/plugin uninstall`.

**Size:** XL (~2000+ LoC: manifest schema + registry client +
sign/verify + install/update/uninstall + tests + docs + the official
registry repo skeleton).

**Why 0.5.0 and not earlier.** The federation work in 0.4.0 (A5
gateway hardening, scoped tokens) is the natural moment to also
introduce plugin trust boundaries — both touch "what's safe to install
across CLIs in a shared environment." Shipping marketplace before
gateway hardening would mean a plugin install on one CLI silently
affects another in the same userId pool, with no scoping.

## 3. `claude-to-deerflow`-style cross-harness skill

**Source:** [`deer-flow/skills/public/claude-to-deerflow/`](../openSrc/deer-flow/skills/public/claude-to-deerflow/) — a working example of an agent harness driving *another* agent harness via slash commands.

**Today.** BrainRouter's `spawn_agent` orchestrates internal child
agents. It can't drive a sibling Claude Code / Codex / Cursor session.

**Change.** A `/handoff <kind> <prompt>` skill that:
- Discovers other active sessions via Stage 2 (`session_list`) of
  the federation plan.
- Picks one matching `<kind>` (`claude-code`, `codex`, `cursor`, or
  `*` for first idle peer).
- Translates the prompt into the target harness's native shape (the
  v0.4.2 Stage 5 vendor adapter work — see [`0.4.0.md`](0.4.0.md)).
- Awaits and ingests the response as a memory record.

Effectively this is "what Stage 5 unlocks at the user-facing level"
once the vendor adapters exist.

**Size:** M (~400 LoC, mostly thin glue once Stage 5 adapters exist).
**Depends on:** v0.4.2 Stage 5 cross-vendor delegation.
