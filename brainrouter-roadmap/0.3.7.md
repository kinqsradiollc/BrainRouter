# 0.3.7 Roadmap — Quick wins after 0.3.6 ships

**Theme.** Small, independent items that don't justify their own minor
release individually but make sense as a coherent "polish + parity"
follow-up to 0.3.6. Each one is sized **S–M** (≤ 1-2 days). No
breaking changes.

**Lineage.** Items 1–3 came out of the peer-CLI parity audit against
Claude Code, Codex, and Antigravity. Items 4–5 came out of the
[`deer-flow/`](../openSrc/deer-flow/) (ByteDance super-agent harness)
and [`semble/`](../openSrc/semble/) (MinishLab CPU-only code-search
MCP) audits.

## Status (as of branch `release/0.3.7`)

- **Item 6 — Terminal UI redesign + in-terminal config wizard:**
  shipped + **scope expanded**. Originally the cycle headline was the
  wizard + `/config` + `/login`; while delivering it we hit the
  raw-stdout redraw wall (Hotfix #1–#2), rebuilt the picker on
  Ink + React (Hotfix #3), and then closed the last surface — the
  regular **chat REPL** — on Ink too (Hotfix #4). That last piece
  was originally a 0.3.8 target; pulling it forward avoided shipping
  a split aesthetic where picker chrome was Ink but the everyday
  turn loop was readline. See
  [`../brainrouter-changelog/0.3.7.md`](../brainrouter-changelog/0.3.7.md)
  for the per-hotfix detail.
- **CLI/server env separation:** shipped on `release/0.3.7`. The CLI
  no longer reads any `.env` file (`brainrouter-cli/.env` /
  `brainrouter/.env` fallback both removed); `~/.config/brainrouter/config.json`
  is the sole credential store. MCP child stderr is piped so server
  logs no longer bleed above the Ink banner. Wizard Skip clears
  `activeServer` instead of silently re-spawning an MCP child from a
  stale profile.
- **Item 7 — CLI Multi-Agent Phase 1:** added late to the cycle as a
  small additive item (~M total). See §7 below. Tracks the spec
  [`../FEATURE_CLI_MULTI_AGENTS_LOGIC_ENHANCEMENT.md`](../FEATURE_CLI_MULTI_AGENTS_LOGIC_ENHANCEMENT.md)
  + tasks [`../FEATURE_CLI_MULTI_AGENTS_LOGIC_ENHANCEMENT_TASKS.md`](../FEATURE_CLI_MULTI_AGENTS_LOGIC_ENHANCEMENT_TASKS.md)
  (local-only file).
- **Items 1–5 (quick wins):** still pending. Cron `/schedule`,
  `/release-notes`, hooks JSON authoring doc, deer-flow Strict
  Tool-Call Recovery, per-vendor MCP install snippets. Cut for the
  0.3.7 ship to keep the cycle focused on the Ink migration; carried
  into 0.3.8.

## 1. Cron-style `/schedule` *(extends today's interval-only `/loop`)*

**State of play.** [`loopRunner.ts`](../brainrouter-cli/src/runtime/loopRunner.ts)
already implements `parseInterval('5s' | '10m' | '1h')` for `/loop`,
but interval-only doesn't cover "every weekday at 9am" or "tomorrow
afternoon" — the canonical scheduled-task use cases. Claude Code's
`/schedule` + `CronList` and Antigravity's Scheduled Tasks both ship
the cron-shape API.

**Scope.**

- **`parseSchedule(raw)`** added next to `parseInterval` in
  [`loopRunner.ts`](../brainrouter-cli/src/runtime/loopRunner.ts).
  Accepts cron expressions (`0 9 * * 1-5`) AND a small set of natural
  absolute targets (`tomorrow 9am`, `2026-06-01T09:00`, `next monday`).
  Returns the next-fire `Date`, or `undefined` for unparseable input.
- **Persisted schedules** at
  `<workspace>/.brainrouter/cli/schedules.json` — each entry carries
  `{ id, kind: 'cron' | 'absolute', expr, prompt, createdAt, lastFiredAt }`.
  On session start, the REPL re-arms any schedules whose next-fire
  has passed since the CLI was last open (catches up overdue runs
  exactly once, then resumes the normal cadence).
- **New slash commands** in
  [`brainrouter-cli/src/cli/commands/obs.ts`](../brainrouter-cli/src/cli/commands/obs.ts):
  - `/schedule add <cron-or-time> <prompt>` — register a new schedule.
  - `/schedule list` — show all registered schedules with next-fire times.
  - `/schedule cancel <id>` — remove a schedule.
- **`/loop` stays** for interval-only use; `/schedule` is the new
  cron / absolute-time entry point.
- **Tests.** Mock `Date.now`, assert `parseSchedule` handles each
  natural-language input shape; round-trip schedule JSON; the
  reentry-after-restart catchup fires exactly once per missed window.

**Size:** M (~300 LoC + tests). **Reference:** Claude `/schedule` +
`CronList`, Antigravity Scheduled Tasks.

## 2. `/release-notes` in-CLI

**State of play.** Users currently have to `cat CHANGELOG.md` or
browse GitHub to see what's new in the version they're running.
Claude Code's `/release-notes` is a trivial parse-and-render. We
already ship marked-terminal for `/help` rendering.

**Scope.**

- Read [`brainrouter-changelog/<current-version>.md`](../brainrouter-changelog/),
  resolving `<current-version>` from `package.json`. Fall back to root
  [`CHANGELOG.md`](../CHANGELOG.md) `[Unreleased]` section if the
  per-version file is absent.
- Render via the existing `marked-terminal` pipeline (no new deps).
- One-screen output; pages with `q` if longer than terminal rows
  (existing pager hook in `/help`).

**Size:** XS (~50 LoC + 2 tests). **Reference:** Claude
`/release-notes`.

- One paragraph in [`README.md`](../README.md) or `SETUP.md` linking
  the schema docs.

**Size:** XS doc-only. **Reference:** Antigravity JSON hooks.

## 4. "Strict Tool-Call Recovery" pattern *(from deer-flow)*

**State of play.** When OpenAI-compatible reasoning models get
malformed tool histories (forced-stop mid-batch, provider-injected
metadata, dangling tool calls that never received a result) they
choke and refuse to continue. We have try/catch around tool calls in
[`agent.ts`](../brainrouter-cli/src/agent/agent.ts), but the recovery
is "throw and let the user retry."

**Scope.**

- **Strip provider metadata on forced-stop.** Before re-injecting
  the chat history on the next turn, scrub provider-internal fields
  (`finish_reason: 'tool_calls'` without follow-up, OpenAI's
  `refusal` blocks on partial completions) that reasoning models
  treat as authoritative even when the user already aborted.
- **Inject placeholder tool results** for any `tool_call` in the
  history that lacks a corresponding `tool` message. Use a
  short envelope like
  `{ "ok": false, "error": "tool call aborted before completion; safe to ignore" }`
  so the model sees a complete request/response cycle even after a
  ^C mid-batch.
- **Detection happens** in `ensureWellFormedHistory(messages)` —
  called at the top of `runTurn` once and before any compaction pass.
- Reference: [`deer-flow/backend/packages/harness/deerflow/runtime/`](../openSrc/deer-flow/backend/packages/harness/deerflow/runtime/)
  — their "Strict Tool-Call Recovery" pattern is the cleanest
  open example we've found.

**Size:** S–M (~200 LoC + targeted tests). **Why pull this in
early.** Reasoning-model adoption (gpt-5, o-series, Claude
extended-thinking, DeepSeek-V4) is rising fast; the symptom is
visible to *every* user once it hits them.

## 5. Per-vendor MCP install snippets *(from semble)*

**State of play.** [`brainrouter/README.md`](../brainrouter/README.md)
documents one install path that branches inline ("for Claude Code…
for Cursor… for Codex…"). Semble's
[`src/semble/agents/{claude,copilot,cursor,gemini,kiro,opencode}.md`](../openSrc/semble/src/semble/agents/)
is the cleanest pattern we've seen — one short, focused file per
vendor with the exact JSON snippet that goes in that vendor's MCP
config.

**Scope.**

- Create `brainrouter/docs/install/<vendor>.md` for: `claude-code`,
  `cursor`, `codex`, `gemini-cli`, `opencode`, `windsurf`. Each is
  ~30 lines: copy-paste JSON snippet keyed by per-user API key + a
  one-paragraph troubleshoot section.
- Cross-link from the main install README.
- Pairs cleanly with the v0.4.0 federation Stage 1 plan (which calls
  for per-client install docs anyway) — this folds the docs work
  forward by one release.

**Size:** S doc-only. **Reference:** semble per-agent install snippets.

## 6. Terminal UI redesign + in-terminal config wizard *(headline)*

**State of play.** A first-run user has to leave the REPL three times
to get a working setup: `brainrouter login` → `brainrouter config` →
`brainrouter` (the chat command). Each of those is an `inquirer`-style
text prompt sequence outside the REPL. Inside the REPL, settings are
scattered across `/theme`, `/personality`, `/effort`, `/mode`,
`/statusline`, `/raw`, `/quiet`, `/model`, `/mcp` — discoverable only
via `/help`. Peer CLIs (Claude Code, Codex, Grok-CLI, DeepSeek-TUI)
put first-run **inside** the REPL as a render-time modal and surface
every knob through a single `/config` panel.

**Scope.** Full spec: [`docs/specs/0.3.7-terminal-ui-redesign.md`](../docs/specs/0.3.7-terminal-ui-redesign.md).

- **Picker primitive extensions** in
  [`cliPrompt.ts`](../brainrouter-cli/src/cli/cliPrompt.ts) — add
  `onCursorChange(index)` and `prefilledOther` so live-preview and
  env-derived defaults work.
- **`/init` wizard** — `Welcome → Theme → Provider → ApiKey →
  Model → MCP → AgentMd → Done` state machine. Auto-fires on REPL
  start when no config exists.
- **`/config` settings home panel** — picker over every CLI knob;
  sub-pickers reuse the wizard's primitives. Verb-overloaded:
  bare opens the panel, `/config <key>` shows, `/config <key>
  <value>` sets.
- **`/login` slash command** — opens the MCP profile editor in-REPL
  (transport picker → fields → reachability test → save).
- **Docs rewrite** — [`brainrouter-docs/cli.md`](../brainrouter-docs/cli.md)
  and [`brainrouter-docs/configuration.md`](../brainrouter-docs/configuration.md)
  lead with the in-REPL flow; env-var matrix moves under
  "Advanced — hand-edit knobs". [`SETUP.md`](../SETUP.md) §2A
  collapses to "run `brainrouter`; the wizard takes over".

**Size:** L (~1.5k LoC + tests + docs). **Reference:**
[`openSrc/claude-code/CHANGELOG.md`](../openSrc/claude-code/CHANGELOG.md)
(`/config` tabbed panel, `/theme`),
[`openSrc/codex/codex-rs/tui/src/onboarding/`](../openSrc/codex/codex-rs/tui/src/onboarding/)
(Step state machine, ApiKeyInputState),
[`openSrc/DeepSeek-TUI/crates/tui/src/tui/onboarding/`](../openSrc/DeepSeek-TUI/crates/tui/src/tui/onboarding/)
(onboarding marker file, verb-overloaded `/config`),
[`openSrc/grok-cli/src/ui/app.tsx`](../openSrc/grok-cli/src/ui/app.tsx)
(render-time ApiKeyModal as onboarding).

## 7. CLI Multi-Agent Phase 1 — registry foundations *(add-on)*

**State of play.** Today the 5 CLI roles (`explorer`, `architect`,
`reviewer`, `worker`, `verifier`) are hardcoded in
[`brainrouter-cli/src/orchestration/roles.ts`](../brainrouter-cli/src/orchestration/roles.ts).
There's no way to add a specialist agent without editing TypeScript;
no tier/depth gates either — a worker could theoretically spawn a
worker that spawns a worker. The peer-CLI audit
([`openSrc/openhuman/src/openhuman/agent/`](../openSrc/openhuman/src/openhuman/agent/),
[`openSrc/claude-code/plugins/`](../openSrc/claude-code/plugins/))
makes the gap obvious — both ship data-driven agent registries with
tiered delegation.

**Scope (small, additive, backward-compatible).**

- **P1.1 — Data-driven agent registry.** New
  `brainrouter-cli/src/orchestration/agentRegistry.ts` + one JSON
  file per built-in under `brainrouter-cli/agents/<id>.json`. Loader
  precedence: workspace > user-global > built-in. `spawn_agent` gains
  optional `agentId` alongside the existing `role` param. New
  `/agents defs` subcommand lists definitions with their source path.
- **P1.2 — Spawn hierarchy + depth caps.** Add `tier: 'chat' |
  'reasoning' | 'worker'` to the definition schema. `chat` parents
  can delegate to `reasoning` / `worker`; `reasoning` can delegate
  to `worker`; `worker` cannot delegate. Default `MAX_SPAWN_DEPTH = 3`
  (overridable via `BRAINROUTER_MAX_SPAWN_DEPTH`).
- **Docs.** New "Custom agents" subsection in
  [`brainrouter-docs/cli.md`](../brainrouter-docs/cli.md) covering
  registry override paths + tier rules.

**Behavior change:** **none** for existing users. All current
`spawn_agent({role:"explorer", …})` calls keep working. Workspaces
without `.brainrouter/agents/` see no change.

**Size:** M (~400 LoC across loader + 5 JSON files + tier check +
tests). **Reference:**
[`openSrc/openhuman/src/openhuman/agent/registry.rs`](../openSrc/openhuman/src/openhuman/agent/registry.rs)
(definition shape, source precedence),
[`openSrc/claude-code/plugins/code-review/`](../openSrc/claude-code/plugins/code-review/)
(per-pack agent layout — the shape packs will adopt in 0.4.2).

**Why land in 0.3.7 (vs deferring to 0.4.0):** the registry is the
foundation every other multi-agent feature builds on. Shipping it as
a small, behavior-neutral addition now lets the 0.4.0 cycle start
with the data structure already in place; otherwise 0.4.0 would have
to fit the loader + every other Phase 2/3 item.

## 8. API key polish *(blocks 0.3.7 ship)*

**State of play.** A pre-ship audit of the wizard / `/login` /
`/config` / `/model` flows turned up three bugs that silently break
provider switching:

1. **`/config provider <id>` reuses the old API key.** The setter at
   [`brainrouter-cli/src/cli/commands/config.ts:534-552`](../brainrouter-cli/src/cli/commands/config.ts)
   hardcodes `apiKey: ctx.config.llm?.apiKey ?? ''` when changing
   provider — so a user with OpenAI saved who runs
   `/config provider deepseek` ends up with an OpenAI key pointed at
   the DeepSeek endpoint, hit by 401 on next turn with no clear
   message. The interactive `editLlm()` flow does this right
   (re-prompts via picker); the bare-arg setter does not.
2. **`/login` only handles MCP transport.** No LLM credential entry
   path — users who want to refresh an API key end up needing
   `/config` or `/init`.
3. **`loadConfig` env fallback misses provider-specific keys.** At
   [`config.ts:84-92`](../brainrouter-cli/src/config/config.ts)
   we only backfill from `OPENAI_API_KEY` / `BRAINROUTER_LLM_API_KEY`.
   A user with DEEPSEEK_API_KEY in their shell + an empty
   `config.llm.apiKey` field gets nothing.

**Scope.** All three fixed in one PR.

- Convert the `provider` setter in `KEY_HANDLERS` to async, open
  the existing API-key prompt (`promptText` with provider-specific
  validate + env prefill) before saving. Pre-fill from the new
  provider's `envKey`; if blank and provider is cloud, reject.
- Add an LLM credential step to `/login` (reuse `editLlm()` from
  `config.ts`) after the existing MCP transport block. Step is
  skippable.
- Expand `loadConfig` to match the saved endpoint against
  `PROVIDER_CATALOG` and fall back to that provider's `envKey`
  before the generic vars.

**Size:** S–M (~150 LoC + tests). **Reference:** the wizard's API-key
step at [`wizard/runner.ts:231-264`](../brainrouter-cli/src/cli/wizard/runner.ts)
which already does this right and is the pattern to copy.

## 9. `/model` quick-swap picker *(blocks 0.3.7 ship)*

**State of play.** `/model` (no arg) just prints the current model
name; users have to type the model id by hand to switch. Both the
wizard's onboarding model step and BrainRouter peer CLIs (Claude
Code, Codex, DeepSeek-TUI) all use an arrow-key picker populated by
the LLM endpoint's `/models` response — but the in-REPL `/model`
command was never wired to that pattern.

**Scope.**

- Extract the wizard's model-step logic
  ([`wizard/runner.ts:267-323`](../brainrouter-cli/src/cli/wizard/runner.ts))
  into a reusable `selectModel(provider, apiKey, endpoint)` helper
  next to `fetchOpenAiCompatibleModels` in
  [`wizard/modelsApi.ts`](../brainrouter-cli/src/cli/wizard/modelsApi.ts).
- Rework `/model` in
  [`commands/ui.ts:176-187`](../brainrouter-cli/src/cli/commands/ui.ts):
  no-arg opens the picker; `<name>` arg keeps the current direct
  switch (script-friendly).
- Picker default-cursors on the currently active model. Live `/v1/models`
  fetch first (5s timeout); fall back to provider static catalog on
  failure with a "(offline catalog)" badge.
- On select: `agent.setModel(model)` + persist to `config.llm.model`
  via `saveConfig`. No restart.
- Detect cross-provider picks gracefully (user picks
  `claude-3-5-sonnet` while configured for OpenAI) and warn —
  recommend `/config provider` to switch endpoints.

**Size:** M (~200 LoC + tests). **Reference:** the Ink picker
primitive in
[`cli/ink/runPicker.tsx`](../brainrouter-cli/src/cli/ink/runPicker.tsx)
+ `pickFromList` callers in
[`commands/config.ts`](../brainrouter-cli/src/cli/commands/config.ts).
