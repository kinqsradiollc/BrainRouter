## Items 1–8

### 1. Goal-leakage across sessions (✅ shipped in PR #26)

**Symptom (reported by user).** Open a new CLI session, type `/goal …`,
get a `GoalConflictError` showing a goal from a *previous* session in the
same workspace:

```
brainrouter[shell]> /goal let's discover our brainrouter
⚠️  A goal is already active:
     explore the whole codebase
     2/unlimited iterations used
Replace it with the new objective? (y/N)
```

**Root cause.** `readGoal` falls back to a workspace-level CLI state file
when the session-scoped `goal.json` doesn't exist yet
([`goalStore.ts:173-176`](brainrouter-cli/src/state/goalStore.ts:173)):

```ts
const legacyPath = getCliStateFile(workspaceRoot, 'goal.json');
if (fs.existsSync(legacyPath)) {
  return normalize(readJsonFile<Partial<Goal> | null>(legacyPath, null));
}
```

So even though new sessions get a fresh sessionKey, every `readGoal`
inside that session finds and returns the OLD workspace-scoped file
written by an earlier session. Effective scope today is **per-workspace,
not per-session** — opposite of what the API name suggests and what the
docs claim.

**Fix.**

- **Stop reading the legacy path on new sessions.** When a sessionKey is
  passed, `readGoal` should return `null` if the session-scoped file
  doesn't exist — never silently fall back. The legacy path remains for
  the no-sessionKey branch (purely for backwards-compat reads from very
  old installs that never had session-scoped goals).
- **One-shot migration.** On the FIRST `setGoal` call of any new
  session, if a legacy workspace `goal.json` exists, *rename* it to
  `<workspace>/.brainrouter.migrated/legacy-goal-<timestamp>.json`
  instead of leaving it where future sessions can pick it back up.
- **Add a test.** "Setting a goal in session A, opening session B,
  calling `readGoal(B)` returns null." Currently this test would fail.

**Affected callers to double-check after the fix:**

- `/goal` slash handler in
  [`workflow.ts:373+`](brainrouter-cli/src/cli/commands/workflow.ts:373)
- The eager-init path in
  [`agent/agent.ts`](brainrouter-cli/src/agent/agent.ts) that bootstraps
  the session key before `runTurn`.
- Goal-continuation loop in
  [`cli/repl.ts`](brainrouter-cli/src/cli/repl.ts) (the `setTimeout`
  re-arm after a turn completes).

### 2. CLI shell redesign (✅ shipped in PR #27)

**Symptom.** Startup output was technical noise: a Node
`ExperimentalWarning` from `node:sqlite` followed by two free-form
`console.log` lines (`Workspace: …`, `Connecting to MCP server profile
"local-http"…`) and then a green `Successfully connected!` — all of
which scrolled real content off-screen on short terminals. Once the
REPL was up there was no visible state: no current workflow, no goal
status, no session id, no model. Users had to chain `/goal status`,
`/workflows`, `/agents`, `/config` separately just to orient.

**Fix.** Seven sub-deliverables landed together as one PR (the surfaces
all touched the same REPL chrome so splitting would have produced
churn for no review-quality gain):

- **Boxed startup banner.** New pure renderer
  [`brainrouter-cli/src/cli/banner.ts`](brainrouter-cli/src/cli/banner.ts)
  draws a single Unicode box with workspace + short-hash, MCP profile +
  transport + online/offline, current workflow (if bound), goal status
  + budget, session prefix, model. Sections silently omit when empty so
  a fresh workspace renders only the rows that apply. The three
  pre-banner `console.log` lines (workspace / connecting / success)
  were deleted — `/workspace` exposes the launch CWD + detection reason
  on demand, and the banner itself IS the success signal.
- **`/where` single-screen state.**
  [`brainrouter-cli/src/cli/whereView.ts`](brainrouter-cli/src/cli/whereView.ts)
  gathers workspace, active workflow + meta, goal status + budget, plan
  items, recent recall (with priorities), and live child sessions, then
  renders themed sections. Empty sections drop out, so the output stays
  proportional to actual state.
- **Persistent status line.** New
  [`brainrouter-cli/src/cli/statusline.ts`](brainrouter-cli/src/cli/statusline.ts)
  centralizes segment renderers. Added `workflow`, `goal`, `plan`, `pr`
  alongside the existing `mode`, `model`, `tokens`, `session`,
  `branch`, `dirty`. `/statusline` now validates against the shared
  `SEGMENT_NAMES` constant — adding a new segment is one switch case,
  not three edits across REPL + handler + help text.
- **`--quiet` + `/quiet` toggle.** Persisted `quiet: boolean` in
  preferences plus a `BRAINROUTER_QUIET=1` env override gate
  tool-start chrome, tool-success summaries, briefing / capture /
  citation memory events, file-mention attachment lines, and tool
  previews. Failures and memory contradictions still surface.
- **Theme module.**
  [`brainrouter-cli/src/cli/theme.ts`](brainrouter-cli/src/cli/theme.ts)
  with semantic tokens (primary, secondary, success, warning, danger,
  info, muted, dim, heading, plain) and three palettes (`dark` —
  original Midnight Ledger; `light` — darker accents for white
  terminals; `mono` — identity for screenshots and pipes). Selection:
  `BRAINROUTER_THEME` env > preference > `dark`. The readline prompt
  and the access-mode accent (read→success, write→primary,
  shell→danger) now consume theme tokens, so `light`/`mono` actually
  reach the surface the user stares at most. Remaining hard-coded
  `chalk.green`/`red` calls in tool-feedback chrome (✓/❌, plan marks,
  error messages) intentionally left for an incremental follow-up.
- **Idle help hint.** 30s after the prompt appears (or after each turn
  ends), if no input arrived and no continuation is running, the REPL
  prints one nudge: "Tip: press `?` or `/help` for commands, `/where`
  for current state." Fires once per session; bare `?` is now wired to
  `/help` so the tip actually works.
- **Node warning suppression — via a CJS bin shim.** The first attempt
  installed the warning filter at the top of `src/index.ts` and didn't
  work: ESM hoists every `import` (including `import 'node:sqlite'`
  transitively through `config/config.ts`) above any top-level code in
  the file, so the warning fires before the filter is registered. The
  fix is a tiny CJS entrypoint at
  [`brainrouter-cli/bin/cli.cjs`](brainrouter-cli/bin/cli.cjs) that
  installs a filtered `warning` listener and `process.emitWarning`
  override synchronously, THEN dynamic-imports the ESM CLI. `package.json`
  `bin` was repointed at the shim; `src/index.ts` keeps a defense-in-depth
  filter for `tsx`/dev runs that bypass the shim.

Also bundled (independent but small and already accounted for in
CHANGELOG): `/tokens` no longer sums every child ever spawned in the
workspace — child usage is now filtered by `parentSessionKey ===
agent.sessionKey`, and `/resume` / `/fork` zero the in-process parent
counters since the persisted transcript doesn't carry per-call usage.

**Out of scope for 0.3.6 (deferred):**

- A full TUI (split panes, scrollback regions, persistent sidebar).
  That's a 0.5.x candidate — would benefit from referencing
  [`antigravity-cli/`](openSrc/antigravity-cli/) for ergonomic patterns
  before committing.
- Mouse support / clickable links.
- Broader theme sweep across the remaining tool-feedback chrome in
  `repl.ts` (per the Tasks.md "consolidation is opt-in and can land
  incrementally" note). Tracked for a follow-up.

### 3. Multi-workflow concurrency

**State of play.** The storage layer already supports many workflows
on disk: every `/feature-dev`, `/spec`, `/review`, `/implement-plan`
creates a new `<workspace>/.brainrouter/workflows/<slug>/` folder with
its own `meta.json`, and `listWorkflows()` returns the full set
([`workflowArtifacts.ts:120`](brainrouter-cli/src/state/workflowArtifacts.ts:120)).
What's missing is the runtime UX to actually *work on more than one at
a time*:

- **`current-workflow.json` is a single pointer per workspace** and
  every `createWorkflow()` silently overwrites it; there's no command
  to switch back to a previously-active workflow without making a new
  one.
  ([`workflowArtifacts.ts:134`](brainrouter-cli/src/state/workflowArtifacts.ts:134))
- **Goal state is workspace-scoped** (see item 1 above) — the same
  `goal.json` is read across every workflow in the workspace. Even
  after the item-1 fix moves it to per-session, that's still not
  per-workflow.
- **`isProcessing` lock serializes turn execution per CLI process.**
  Even if multiple goals existed, only one continuation loop can run
  at a time inside a single REPL.
  ([`cli/repl.ts:249`](brainrouter-cli/src/cli/repl.ts:249))

**Scope for 0.3.6.**

- **`/workflow switch <slug>`** — refocus on an existing workflow.
  Updates the current pointer; loads the workflow's saved goal state
  into the session (see next bullet). Refuses if the source workflow's
  goal is `active` and would conflict with the target's `active`
  state — the user must pause/complete the source first.
- **Per-workflow goal binding.** Move goal persistence to
  `<workflow>/goal.json` so switching workflows carries the goal with
  it. When no workflow is bound (quick-task path), fall back to the
  session-scoped path from item 1. Migration: on a `/workflow switch`,
  the current session's goal is moved into the target workflow's
  folder.
- **`/workflows` upgrade.** Show per-workflow goal status alongside
  the spec/tasks/walkthrough markers, plus iteration budget usage.
  Makes the list a real switcher dashboard, not just a folder index.
- **`/workflow pause` + `/workflow resume <slug>`** — pause the
  current workflow's goal (already supported as `/goal pause`) and
  provide a one-shot resume-by-slug. The latter combines
  `/workflow switch` + `/goal resume`.
- **Explicit conflict surface on `createWorkflow()`.** When
  `/feature-dev` or `/spec` would clobber an in-progress workflow's
  current pointer, prompt before flipping — same UX shape as
  `GoalConflictError`. Users should never lose track of a workflow
  because they typed a new `/spec` while another was open.

**Out of scope for 0.3.6** (deferred to 0.4.0 federation or later):

- **Concurrent goal execution within one CLI process.** The
  `isProcessing` lock stays. Two parallel continuation loops in the
  same REPL would require a far bigger redesign (multiplexed turn
  queue, per-workflow LLM-semaphore slice, UI for "which workflow am
  I watching"). Easier to ship "multiple CLI windows, one per
  workflow" via the v0.4.0 federation plan below.
- **Cross-workflow handoff** — covered by Stage 4 of v0.4.0.

### 4. Mid-turn user-choice prompt — `ask_user_choice` (✅ shipped in PR #30)

**Symptom (pre-fix).** brainrouter-cli had `askYesNo`
([`cliPrompt.ts`](../brainrouter-cli/src/cli/cliPrompt.ts)) for one-shot
binary confirmations and used `inquirer` in
[`index.ts`](../brainrouter-cli/src/index.ts) for the one-time
`brainrouter config` setup wizard — but **no multi-choice mid-turn
prompt** the agent could call to pause and present 2–4
mutually-exclusive options to the user before committing to a path.
When the agent reached an inflection point with multiple defensible
approaches, it had two bad options: silently pick one, or burn a turn
asking in prose and waiting for the user's free-form reply.

**Fix — what shipped.** A new local tool `ask_user_choice` registered
alongside `run_command` / `read_file` / etc., with an **interactive
arrow-key picker** that goes beyond the original line-mode spec to
match Claude Code's `AskUserQuestion` UX:

- **`askChoice(question, options, opts)` helper** at
  [`brainrouter-cli/src/cli/cliPrompt.ts`](../brainrouter-cli/src/cli/cliPrompt.ts).
  Pure reducer (`initPickerState` / `reducePicker` / `renderPicker`) split
  out from the side-effecting orchestrator so the keystroke→state→render
  loop is unit-testable without faking a TTY.
- **Arrow-key picker UX.** ↑/↓ navigate, ENTER confirm; SPACE toggles
  in `multiSelect` mode; q / Esc / Ctrl+C cancel (surfaces as
  `CancelledChoiceError`). Cursor hidden during render; in-place
  redraw on every keystroke.
- **Always-on synthetic "Other" row.** Appended to every prompt. Picking
  it drops to a free-text input phase (Backspace edits, Esc bails back
  to the picker, ENTER finalizes with the typed string replacing the
  "Other" label). The user is never trapped between bad options.
- **Eager input validation** before any screen writes: 2–4 options
  required, duplicate labels rejected case-insensitive,
  `"Other"` reserved (the synthetic free-text row owns it), each
  option needs both `label` and `description`.
- **Non-TTY behavior is strict.** Throws `NoTTYError` in CI / piped /
  `brainrouter run` instead of silently defaulting to option 1.
  Silent child agents (`spawn_agent --silent`) get the same error
  with a clearer "decide yourself and report which option you picked"
  message.
- **Tool registered with JSON schema** at
  [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts)
  (read tier — interaction primitive, no workspace side effects).
  Signature:
  ```
  ask_user_choice({
    question: string,
    header: string,                                          // ≤12 chars chip
    options: Array<{ label: string; description: string }>,  // 2–4 items
    multiSelect?: boolean                                    // default false
  }) → { answer: string | string[] }
  ```
- **`pickerActive` flag** gates the REPL's `shift+tab` access-mode
  handler so the cycle can't fire mid-picker.
- **System-prompt rule** at
  [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts)
  scopes the tool tightly: only for genuine ambiguity with 2–4
  mutually-exclusive reasonable approaches. Explicit anti-patterns
  ruled out: trivial confirmations (`askYesNo` covers those), things
  the agent can decide itself, substitute-for-thinking, and shifting
  load-bearing decisions back to the user when the spec / files
  already imply the answer.

**Latent bug fixed along the way.** A stray `setRawMode(false)` at REPL
startup (since the very first CLI commit) had been silently disabling
raw mode that `readline.createInterface` enables automatically for a
TTY. Symptoms: Backspace at the `brainrouter[shell]>` prompt echoed
`^?` instead of erasing; shift+tab access-mode cycling never fired.
The bug was latent until the picker work exposed it. Removed plus
belt-and-suspenders forcing of `terminal: true` + explicit
`emitKeypressEvents` + `setRawMode(true)` at REPL startup to handle
terminal-multiplexer environments where readline's auto-init is
unreliable (tmux, screen, VS Code integrated terminal, certain SSH
setups — they return `undefined` for `input.isTTY`).

The raw-mode story took 5 fix commits to land cleanly. The deepest
insight was discovering that `process.stdin.setRawMode(flag)` has a
side effect beyond toggling the mode: it also resets
`this.readableFlowing = null` on the stream. A redundant external
`setRawMode(true)` call right after `rl.resume()` (which already
calls it internally for terminal interfaces) puts the stream back
in "auto" flowing state with no follow-up kick — the prompt renders
but keystrokes never reach readline. The committed code documents
this at every guard point so future maintainers don't reintroduce
the bug.

**Tests.** 19 new tests in
[`brainrouter-cli/src/agent.test.ts`](../brainrouter-cli/src/agent.test.ts)
covering: init shape (synthetic "Other" appended), navigation
(down/up wrap), single-select ENTER, multi-select SPACE+ENTER +
empty-selection guard, Other transition + typed finalization,
Backspace + Esc behavior in Other phase, q + Ctrl+C cancellation
paths, renderer cursor/checkbox glyphs, single vs multi-select
footer hints, eager validation (2–4 length, duplicate labels,
reserved "Other"), and the non-TTY guard. Total brainrouter-cli
suite: 141 → 160 (baseline pre-2b was 141; PR landed at 160).

**Docs.**
[`brainrouter-docs/cli.md`](../brainrouter-docs/cli.md) gained a new
"Mid-turn user prompts" section covering both `askYesNo` (previously
undocumented at the surface level) and `ask_user_choice` — picker
walkthrough, behaviour table, schema, when-to-call rule. Plus a
new row in the Local tools table. CHANGELOG updated under Features
in both [`brainrouter-changelog/0.3.6.md`](../brainrouter-changelog/0.3.6.md)
and the root [`CHANGELOG.md`](../CHANGELOG.md) `[Unreleased]`
section.

**Out of scope — deferred.**

- **Surfacing the prompt in non-CLI surfaces** (dashboard chat, HTTP
  MCP clients). The local-tool design keeps this CLI-only; remote
  clients attached to the same MCP wouldn't see it. Federation
  Stage 3 (v0.4.0) is where cross-process user prompts would land
  if we want them.
### 5. Structured reasoning-step capture in working memory (✅ shipped in PR #32)

**Symptom (pre-fix).** BrainRouter's working memory
([`brainrouter/src/memory/working/`](brainrouter/src/memory/working/))
captured Current Plan (✅, `tasks.json`) and Tool Outputs (⚠️ opt-in,
via `memory_working_offload` with the default `kind="tool_output"`),
but **Reasoning Steps were nowhere durable** — they lived only as prose
inside assistant messages in the transcript. The system prompt already
told the model to "walk through your reasoning before tool calls when
the task is non-trivial," but that reasoning evaporated into chat
content. The next turn's briefing had the recall block and the working
canvas, but nothing structured about *why* the agent did what it did in
the previous batch. After a long session, `/working` showed the
offloaded tool payloads with no reasoning trail.

The `WorkingStep.kind` field was already free-form on the schema, so
the primitive existed — what was missing was the discipline / guidance
for the agent to emit reasoning steps and the surface for them to land
on visibly.

**Fix — what shipped.** Four sub-deliverables landed together because
they all support the same end-to-end "agent emits 'why' → it lands in
the canvas → next turn sees it" loop; splitting would have produced
churn for no review gain:

- **System-prompt rule** in
  [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts).
  New line under "During the work":
  > **Capture the WHY, not just the WHAT.** After every non-trivial
  > tool batch (≥3 tool calls OR any single tool that returned >2KB),
  > call `memory_working_offload` ONCE with `kind: "reasoning"`,
  > `title: "Why: <short>"`, and a 1-paragraph summary of the DECISION
  > you made and WHY (not what the tools returned). This pairs with
  > the ~1,000-token offload rule above: payload offload is about
  > token budget, reasoning offload is about the audit trail the next
  > turn's briefing will surface back to you.

  Scoped distinct from the existing >1000-token rule on purpose — that
  one is for payloads (what came back), this one is for decisions
  (why). Both can fire on the same batch.

- **Canvas distinct style** in
  [`brainrouter/src/memory/working/canvas.ts`](../brainrouter/src/memory/working/canvas.ts).
  Pre-PR audit: the canvas did NOT render by `kind` — every step
  rendered as an identical label-only node, with only the
  `activeNodeId` fill providing any visual variation. Fixed by emitting
  a `style <nodeId> stroke-dasharray:4 4,stroke:#9f7aea,stroke-width:2px`
  line for every step with `kind === "reasoning"`. Active-node fill
  still wins (style line ordering preserved). Reasoning nodes are now
  visually separable from `tool_output` and `compressed_summary` nodes
  at a glance on `canvas.mmd`.

- **Briefing surface** in
  [`brainrouter-cli/src/memory/briefing.ts`](../brainrouter-cli/src/memory/briefing.ts).
  When `memory_working_context` returns the expected JSON shape, the
  briefing renders the working-memory section structurally instead of
  opaque-dumping a 1500-char slice. Two blocks:
  - `Recent steps:` — the MCP's already-injected `recentSteps` tail
    in order (kind + title + summary preview).
  - `Recent reasoning (why-trail):` — up to 3 most-recent
    reasoning-kind steps that are NOT already in `recentSteps` (so a
    reasoning step still in the tail doesn't double-print).

  The cap-at-3 on the why-trail is deliberate: without it, an agent
  that offloaded reasoning every batch would stuff the next turn's
  briefing with its own past commentary. When the payload isn't
  parseable JSON, the renderer returns null and the existing
  opaque-dump + `redactText` branch takes over — so the secrets-leak
  test in `memory.test.ts` is unaffected.

- **Tests.** Five new tests:
  - **Round-trip** in
    [`brainrouter/src/__tests__/working-memory.test.ts`](../brainrouter/src/__tests__/working-memory.test.ts):
    offload with `kind: "reasoning"`, read back via
    `memory_working_context`, assert the kind survives across `steps`
    AND `state.injectedState.recentSteps`. Guards against a future
    regression that silently defaults the kind back to `tool_output`.
  - **Canvas render** (same file): mixed `tool_output` +
    `reasoning` steps, assert the reasoning node carries the dashed
    `stroke-dasharray` style and the tool_output node does not.
  - **Prompt assembly** in
    [`brainrouter-cli/src/tests/prompt.test.ts`](../brainrouter-cli/src/tests/prompt.test.ts):
    loose-regex pin on the rule's wording (the kind value, the
    `Why:` title convention, the ≥3 / >2KB triggers) so a future
    refactor of `systemPrompt.ts` can't silently delete it.
  - **Briefing structural surface** in
    [`brainrouter-cli/src/tests/memory.test.ts`](../brainrouter-cli/src/tests/memory.test.ts):
    JSON stub with mixed steps; assert both blocks render, dedup
    against `recentSteps` works.
  - **Why-trail cap-at-3** (same file): 8 reasoning steps in, last 3
    render, the older 5 do not.

  Suite totals: `brainrouter` 260 → 262 (+2); `brainrouter-cli`
  164 → 167 (+3).

**Docs.** New
[`brainrouter-docs/cli.md`](../brainrouter-docs/cli.md) subsection
under "Working-memory canvas" — "Reasoning capture (why-trail)" —
covering when the rule fires, what lands in `steps.jsonl`, the
dashed-border canvas convention, the two briefing blocks, and the
cap-at-3 rationale. CHANGELOG updated under Features in both
[`brainrouter-changelog/0.3.6.md`](../brainrouter-changelog/0.3.6.md)
and the root [`CHANGELOG.md`](../CHANGELOG.md) `[Unreleased]` section.

**Out of scope — deferred.**

- **Auto-capture of reasoning** (CLI intercepts assistant prose and
  writes the reasoning step itself, no LLM round-trip). Would require
  parsing assistant content for a reasoning-block convention — more
  brittle and a different tradeoff. Deferred to v0.4.x.
- **Auto-offload of large tool outputs under token pressure.** Related
  but distinct from reasoning offload — see the
  "auto-offload-under-pressure" notes. Skipped from this release on
  user direction.
- **LLM-side compliance with the rule is best-effort.** Smaller / less
  aligned models routinely skip explicit "after every batch, call
  tool X" instructions. The spec acknowledged this — the system-prompt
  rule is steering, not enforcement. The canvas style + briefing
  surface still pay off the moment ANY reasoning step lands (whether
  the model emits it autonomously or a user nudges it).

**Bundled (out-of-scope) fix on the same cycle: PR #34.** Hitting
`/working` during manual verification of this PR surfaced a separate
REPL freeze (every spinner-based slash command and the
`ask_user_choice` picker left readline dead — Backspace echoed `^?`,
ENTER stopped submitting). Root cause: `ora`'s default
`discardStdin: true` drops readline's keypress listener as a side
effect of `stdin-discarder.stop()`. Fixed in
[PR #34](https://github.com/kinqsradiollc/BrainRouter/pull/34) (separate
branch off `main`) by routing every `ora()` call through a new
`brainrouter-cli/src/cli/spinner.ts` factory that always passes
`discardStdin: false`. Verified end-to-end with a standalone Node
repro of the listener-drop. Not part of Item 2c, but worth noting
because the testing flow uncovered it.
### 6. `/grill-me` — clarifying questions before implementing (✅ shipped in PR #31)

**Symptom (pre-fix).** When the user types `/spec <feature>` we kick
off a spec-driven workflow that produces `spec.md` first, then
`tasks.md`, then implementation. But for plain prompts ("add a
`/uptime` slash command that shows REPL uptime") there was no built-in
pause-to-clarify gate — the agent picked a reading of the request and
ran. `askYesNo` covered binary confirmations and item 2b
(`ask_user_choice`, shipped in PR #30) covered multi-choice mid-turn
prompts, but neither helped the agent when its *first read* of the
request was genuinely ambiguous on multiple axes (scope, format,
dependencies). Antigravity's `/grill-me` is the explicit answer:
pause, ask 2–5 questions back, then proceed once the user has resolved
the ambiguity.

**Fix — what shipped.** A user-initiated slash command that primes
the model into CLARIFY mode for one turn, steering it toward
`ask_user_choice` instead of implementation tools:

- **New slash command** `/grill-me [--force] <task>` at
  [`brainrouter-cli/src/cli/commands/workflow.ts`](../brainrouter-cli/src/cli/commands/workflow.ts).
  Registered alongside `/spec` and `/feature-dev`; routes through the
  existing `runAgentTurn` pipeline. Latches
  `agent.activeSkill = 'grill-me'` and calls `refreshSystemPrompt()`
  so the CLARIFY overlay lands in `chatHistory[0]` before the model
  sees the prompt.
- **Skip-if-plan-exists guardrail** via the exported
  `shouldSkipGrillMe(workspaceRoot, force)` helper. When
  `getCurrentWorkflow(workspaceRoot)` returns a slug AND `spec.md`
  exists in that workflow folder, `/grill-me` refuses with:
  > Plan already exists at `<path>`.
  >   Drop into it with `/workflow switch <slug>`, or use
  >   `/grill-me --force` to clarify additional details.

  Cheap protection against re-litigating answers the user already
  gave during the spec phase. `--force` is the explicit escape hatch
  when scope has drifted.
- **System-prompt overlay** at
  [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts).
  Scoped to `context.activeSkill === 'grill-me'`; six lines:
  ```
  ## CLARIFY mode (grill-me)
  - Do NOT make file edits, run shell commands, or spawn worker agents this turn.
  - Ask 2–5 questions to disambiguate scope, format, and unstated assumptions.
  - Prefer `ask_user_choice` for mutually-exclusive options; plain prose for free-form input.
  - (`askYesNo` is a CLI-internal gate the framework triggers — do NOT try to call it as a tool.)
  - End with a one-paragraph "what I'll do once you answer" so the user can sanity-check the read.
  ```
- **Post-turn cleanup.** The REPL's finally block (cli/repl.ts:597)
  clears `activeSkill = undefined` and calls `refreshSystemPrompt()`
  so the CLARIFY overlay disappears from `chatHistory[0]` before the
  user's next plain prompt — otherwise the model would see "do not
  make file edits" carrying over.
- **Discoverability fixes** (Copilot review follow-up). `/grill-me`
  registered in `SLASH_COMMANDS` (tab autocomplete) and in
  `HELP_CATEGORIES.workflow` (so `/help workflow` lists it). Without
  these the command worked but was undiscoverable.

- `prompt.test.ts`: `activeSkill = 'grill-me'` produces a system prompt

- `cli-prompt.test.ts`: skip guard fires when current workflow has

### 7. Review-policy toggle — `/mode` and `/review-policy` (✅ shipped in PR #35)

**Symptom (pre-fix).** BrainRouter already had every primitive needed
for "do I halt and ask before risky things?" — `askYesNo` on
`run_command`, `/yolo on|off` for auto-approve shell, `/approve` for
workflow plans, and the access-mode cycle (read → write → shell). The
problem was that these were *fragmented*: a new user had to learn four
separate switches with different scopes (per-call, per-tool, per-mode,
per-workflow) to express the single intent "run on autopilot" or "ask
me before doing anything destructive." Claude Code's
`permissions.defaultMode` and Antigravity's Artifact Review Policy are
both single switches that consolidate the same primitives.

**Fix — what shipped.** Two new session-level slash commands that
consolidate the existing primitives into one mental model. **No new
framework gates and no new approval halts** — this is pure
consolidation, as the headline says. The behavioral change for
existing users is exactly that `/yolo on` now flips both new axes
under the hood:

- **`/mode planning|fast`** at
  [`brainrouter-cli/src/cli/commands/guard.ts`](../brainrouter-cli/src/cli/commands/guard.ts).
  `planning` (default) routes every `run_command` through the y/N
  prompt; `fast` skips the prompt for the safe set and routes
  dangerous commands through `askYesNo` anyway. Bare `/mode` prints
  current value + the two options + a one-liner on what each
  affects. Persisted as
  `executionMode: 'planning' | 'fast'` in
  [`preferencesStore.ts`](../brainrouter-cli/src/state/preferencesStore.ts).
- **`/review-policy request|proceed`** in the same file. `request`
  (default) preserves today's prose-based "ready for your approval?"
  → user types `/approve` workflow gesture. `proceed` is a
  system-prompt overlay that tells the model to apply multi-file
  changes and report after, without the prose pause. `/approve`
  itself is unchanged — it stays as the explicit per-workflow
  approval gesture. Persisted as `reviewPolicy: 'request' | 'proceed'`.
- **`/yolo` becomes a one-line alias.** `/yolo on` calls
  `applyYoloOn(workspaceRoot)` which writes `executionMode: 'fast'`,
  `reviewPolicy: 'proceed'`, AND `autoApproveShell: true` (legacy
  field mirrored for the alias transition). `/yolo off` calls
  `applyYoloOff` which restores the defaults on all three. Bare
  `/yolo` prints the resolved state plus a nudge toward `/mode` and
  `/review-policy` for finer control.
- **`isDangerousCommand` + pure resolver** at
  [`brainrouter-cli/src/runtime/dangerousCommand.ts`](../brainrouter-cli/src/runtime/dangerousCommand.ts).
  New file holding the single source of truth for what counts as
  destructive (`rm -rf`, `dd`, `chmod 777`, `sudo`, `git push --force`,
  `mkfs`, `kubectl delete`, `curl … | sh`, `DROP TABLE`, `docker
  prune`, …) plus a pure `resolveRunCommandApproval(prefs, command,
  {silent})` that returns `'auto-approve' | 'ask' | 'deny-silent'`.
  Pure so that `run_command` gating logic is unit-testable without
  TTY mocking. The brief said "find the existing helper" — there was
  none, so this *is* the helper.
- **`run_command` gate** at [`agent.ts`](../brainrouter-cli/src/agent/agent.ts).
  Old branching on `prefs.autoApproveShell` replaced with a single
  call to the resolver. Three outcomes route to: auto-approve with
  the existing `▶  Auto-approved` chrome, `askYesNo` with a new
  red `(flagged as potentially destructive)` suffix when the
  resolver said yes-but-dangerous, or `deny-silent` with a clearer
  error for silent children (separate error strings for
  safe-but-no-opt-in vs. dangerous-in-silent-child).
- **System-prompt overlay** (`policyOverlay`) at
  [`systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts).
  Only emits when `executionMode === 'fast'` or
  `reviewPolicy === 'proceed'`. `fast` line tells the model to skip
  the "may I run this?" prose for safe commands; `proceed` line
  tells it to apply multi-file plans without the "ready for your
  approval?" pause. Defaults emit no overlay — the rest of the
  prompt is already tuned for that stance.
- **Migration on read.** Pre-0.3.6 prefs files only had
  `autoApproveShell: boolean`. `migrateLegacyShell()` in
  preferencesStore back-fills `executionMode: 'fast'` +
  `reviewPolicy: 'proceed'` from `autoApproveShell: true` on first
  read of the new fields, leaves the legacy field on disk for
  external readers during the alias transition, and explicitly does
  *not* override when the user has set the new fields explicitly
  (e.g. `autoApproveShell: true` + `executionMode: 'planning'`
  reads back as planning — the user already migrated forward).
- **Statusline surfacing.** New `exec` segment in
  [`statusline.ts`](../brainrouter-cli/src/cli/statusline.ts). Shows
  `fast` only — `planning` is hidden as the default since surfacing
  it would add chrome on every prompt for the vast majority of
  users. Named `exec` (not `mode`) deliberately to avoid colliding
  with the existing access-mode `mode` segment and breaking any
  user `statusline:` pref strings.
- **`/where` surfacing.** Workspace block in
  [`whereView.ts`](../brainrouter-cli/src/cli/whereView.ts) gains a
  third line `exec    <mode>  ·  review <policy>` — shown
  unconditionally regardless of defaults, since `/where` is the
  "tell me everything" surface (unlike the statusline, which hides
  defaults to stay quiet).
- **Discoverability fixes.** `/mode` and `/review-policy` registered
  in `SLASH_COMMANDS` (tab autocomplete) and in
  `HELP_CATEGORIES.guard` next to `/permissions` and `/yolo`. The
  `/statusline` help text was extended to mention the new `exec`
  segment so users discover it by reading the prompt's own help.

**Tests.** 11 new tests across two files. Suite: 164 → 175 in the PR
(and 178 after the merge picked up PR #32's reasoning-step tests).

- [`tests/state.test.ts`](../brainrouter-cli/src/tests/state.test.ts):
  defaults (`planning` + `request`); round-trip; `applyYoloOn` flips
  both axes and keeps the legacy mirror in sync; `applyYoloOff`
  restores defaults; migration from `autoApproveShell: true`
  back-fills the new fields; migration from
  `autoApproveShell: false` leaves defaults; explicit new fields
  override the legacy migration (user-explicit beats heuristic).
- [`tests/runtime.test.ts`](../brainrouter-cli/src/tests/runtime.test.ts):
  `isDangerousCommand` positive cases (`rm -rf`, `dd`, `chmod 777`,
  `sudo`, force-push, `curl|sh`, `mkfs`, `DROP TABLE`, `docker
  prune`, `kubectl delete`); negative cases (`npm run build`, `ls`,
  `rmdir` ≠ `rm`, `git push origin main` ≠ force-push); resolver
  across planning vs fast with regression guard "fast is not
  yolo-everything"; silent-child matrix asserting safe + opt-in =
  auto-approve, safe + no-opt-in = deny, dangerous = deny
  regardless of mode (silent children can't answer y/N).

**Docs.** [`brainrouter-docs/cli.md`](../brainrouter-docs/cli.md)
access-mode section gained a Mode/Policy/Yolo table showing default
and toggled behaviour for both knobs side-by-side. The Guard
sub-table grew rows for `/mode` and `/review-policy` and reframed
the `/yolo` row as "Alias for `/mode fast` + `/review-policy
proceed`." Both changelogs updated under Features in
[`brainrouter-changelog/0.3.6.md`](../brainrouter-changelog/0.3.6.md)
and root [`CHANGELOG.md`](../CHANGELOG.md) `[Unreleased]`.

**Incidental safety improvement.** Dangerous commands in **silent
child** agents are now always denied regardless of mode. Pre-0.3.6,
`/yolo on` let silent children run anything (including `rm -rf /`)
because the silent-child branch only checked `autoApproveShell`,
not `isDangerousCommand`. New resolver enforces both. Worth
calling out because it's strictly a tightening over prior behaviour
— users who relied on silent children running `rm`-style
destructives will see them denied now; the workaround is having a
parent agent run the command, or splitting it into a non-`rm`
equivalent.

**Out of scope — deferred (called out in the PR body).**

- **Auto-clarify pass in planning mode.** Spec § 7 (above the
  rewrite line) said planning mode "triggers a short
  `/grill-me`-style clarify-or-confirm pass" for plain prompts.
  That auto-trigger doesn't exist today (PR #31's `/grill-me` is
  explicitly user-invoked). The recommendation in this PR was
  option (b): defer entirely. `/grill-me` already gives the
  explicit lever; the heuristic for "when is a prompt ambiguous
  enough to auto-clarify?" needs observation data on `/grill-me`
  usage before automating it. Tracked as a possible follow-up for
  0.4.x once that data exists.
- **New halt-before-multi-file-apply gate.** Spec described
  `reviewPolicy: 'request'` as "halts and emits `askYesNo('Approve
  plan?')` before applying." That gate didn't exist today either —
  today's request-style review is prose-based ("ready for your
  approval?") ending in the user typing `/approve`. Adding a new
  framework halt would be a substantial scope expansion for a
  "consolidation, not new capability" PR. Implemented `request` as
  today's behaviour; `proceed` as a prompt overlay relaxing it.
- **Naming drift from spec.** Two small deltas worth flagging:
  spec said the new statusline segment should be named `mode` — but
  there's already a `mode` segment (access mode), so it shipped as
  `exec` to avoid breaking existing `statusline:` pref strings.
  Spec also said `/mode` and `/review-policy` should live in
  `commands/ui.ts` — they shipped in `commands/guard.ts` instead,
  alongside `/yolo` / `/permissions` / `/sandbox` (the natural
  siblings, since they're all approval/permission knobs).

**Merge mechanics.** The PR was rebased onto `main` once after PRs
#32 and #34 landed; one real conflict in
`brainrouter-changelog/0.3.6.md` (both PRs wanted the same Features
slot) and four files auto-merged cleanly

auto-merge correctly preserved PR #32's reasoning-step prompt rule
and PR #34's spinner factory alongside the new commands.
### 8. `/effort low|medium|high` — reasoning depth preference

**State of play.** BrainRouter passes whatever reasoning configuration
the LLM provider supports (or doesn't). There's no first-class
session-level "how hard should the model think" knob, so users on
reasoning-capable backends (gpt-5, o-series, Claude
extended-thinking, DeepSeek-V4) have to repeat themselves in prose
("think carefully", "be terse") every turn to influence depth. Claude
Code's `/effort` and `$CLAUDE_EFFORT` env have been the cleanest
reference shape we've seen.

**Scope for 0.3.6.**

- **`/effort low|medium|high`** slash command in
  [`brainrouter-cli/src/cli/commands/ui.ts`](brainrouter-cli/src/cli/commands/ui.ts).
  Bare `/effort` prints the current level + the three options + a
  one-liner on what each affects.
- **Persisted preference** `effort: 'low' | 'medium' | 'high'`
  (default `medium`) in
  [`preferencesStore.ts`](brainrouter-cli/src/state/preferencesStore.ts).
  Env override: `BRAINROUTER_EFFORT` (matches the `BRAINROUTER_THEME` /
  `BRAINROUTER_QUIET` precedence pattern set in item 2).
- **System-prompt overlay.** A 1-line block in the assembled prompt:
  - `low` → "Be terse. Skip ceremony. One-paragraph answers when the
    user's question fits in one paragraph."
  - `medium` → (no overlay; current behavior)
  - `high` → "Reason step-by-step before acting. Audit your evidence
    against the goal before each tool call."
- **Provider-aware reasoning-config forwarding** in
  [`agent.ts`](brainrouter-cli/src/agent/agent.ts) when the LLM
  endpoint advertises a reasoning slot (OpenAI Responses API
  `reasoning.effort`, Anthropic `thinking`, DeepSeek `thinking_level`).
  Best-effort: map `low/medium/high` to provider-native enums, fall
  through silently when the endpoint doesn't expose one.
- **Surfacing.** New `effort` statusline segment; show in `/where`'s
  Session block.
- **Tests:** preference round-trip; env override precedence over
  preference; system-prompt overlay assembly per level.

**Acceptance:** one slash command + one env var swap reasoning depth
session-wide; both the system prompt and (when supported) the
provider's reasoning slot reflect it; status surfaces the level.
Tests cover the three pref/env/overlay paths.
