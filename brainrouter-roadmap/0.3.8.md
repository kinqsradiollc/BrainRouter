## 1. Runtime child-drain guardrail

**Problem.** The system prompt says to call `wait_agent` /
`wait_agents`, but `agent.ts` currently accepts a no-tool prose answer
as a clean exit even if this turn spawned child agents. That creates
the visible failure mode: "I am waiting..." is just text, so nothing
happens.

**Change.**

- Track child ids spawned during the current parent turn.
- Before accepting a no-tool final answer, inspect those child records.
- If any are `pending` or `running`, do one of:
  - inject a corrective tool-result/system message telling the model to
    call `wait_agents`, then continue the loop, or
  - auto-call `wait_agents` with a bounded timeout and feed the result
    back to the model for synthesis.
- If the timeout expires, return an explicit "children still running"
  response with ids and a suggested `/continue`, not vague silence.
- Reuse the `/continue` logic in `workflow.ts` only as UX wording; the
  invariant belongs in `Agent.runTurn`.

**Files.**

- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter-cli/src/orchestration/tools.ts`
- `brainrouter-cli/src/orchestration/orchestrator.ts`
- `brainrouter-cli/src/tests/agent-runtime.test.ts`

**Acceptance.**

- A model response that says "I am waiting" after spawning children does
  not end the turn while children are still running.
- A broad prompt that spawns children drains or reports pending child ids
  deterministically.
- The loop limit still works and does not spin forever on failed child
  waits.

**Size:** S-M. **Target:** 0.3.8.

## 2. Foreground task vs background delegate semantics

**Problem.** `spawn_agent` is both a foreground task and a background
delegation primitive depending on `wait`. Models do worse when the
tool name does not encode intent.

**Change.**

- Add a blocking `task_agent` wrapper for "do this child task and wait
  for the result".
- Add a background `delegate_agent` wrapper for "send this off and keep
  working".
- Keep `spawn_agent` and `spawn_agents` as low-level compatibility
  tools.
- Update prompt policy so the model chooses: direct answer, direct
  tool, foreground `task_agent`, background `delegate_agent`, later
  0.4.0 typed `delegate_*` tools.

**Files.**

- `brainrouter-cli/src/orchestration/tools.ts`
- `brainrouter-cli/src/prompt/systemPrompt.ts`
- `brainrouter-cli/src/cli/ink/toolFormat.ts`
- `brainrouter-docs/cli.md`
- `brainrouter-cli/src/tests/mcp.test.ts`

**Acceptance.**

- `task_agent` returns completed child output or a timeout envelope.
- `delegate_agent` returns a running child id and clear "continue
  working" semantics.
- `spawn_agent({ wait: true })` remains supported.

**Size:** M. **Target:** 0.3.8.

**Problem.** Child tool activity is mostly a spinner/status-line event.

- Show child id, role, tool name, duration, success/error, and short

- Add a compact "running children" status row when parent is waiting.

**Files.**

## 4. Safe parallel execution for independent tool calls

**Problem.** The runtime executes all tool calls from one assistant
message serially. That is safe, but it slows read-heavy turns and makes
multi-agent starts feel sluggish.

**Change.**

- Keep write/shell operations serialized by default.
- Allow safe read-only local tools and background child spawns to run in
  parallel when emitted in the same assistant message.
- Preserve transcript order with stable call ids.
- Add a kill switch:
  `BRAINROUTER_PARALLEL_SAFE_TOOL_CALLS=false`.

**Files.**

- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter-cli/src/runtime/tools/*`
- `brainrouter-cli/src/tests/agent-runtime.test.ts`

**Acceptance.**

- Two `read_file` calls can execute concurrently.
- `write_file`, `edit_file`, `apply_patch`, and foreground `task_agent`
  remain ordered.
- Tool-result transcript entries still match the correct OpenAI
  `tool_call_id`.

**Size:** M. **Target:** 0.3.8.

## 5. Cron-style `/schedule` *(carried from 0.3.7)*

**Problem.** [`loopRunner.ts`](../brainrouter-cli/src/runtime/loopRunner.ts) already implements `parseInterval('5s' | '10m' | '1h')` for `/loop`, but interval-only doesn't cover "every weekday at 9am" or "tomorrow afternoon" — the canonical scheduled-task use cases. Claude Code's `/schedule` + `CronList` and Antigravity's Scheduled Tasks both ship the cron-shape API.

**Change.**

- **`parseSchedule(raw)`** added next to `parseInterval` in `loopRunner.ts`. Accepts cron expressions (`0 9 * * 1-5`) AND natural absolute targets (`tomorrow 9am`, `2026-06-01T09:00`, `next monday`). Returns the next-fire `Date`, or `undefined` for unparseable input.
- **Persisted schedules** at `<workspace>/.brainrouter/cli/schedules.json`. Each entry carries `{ id, kind: 'cron' | 'absolute', expr, prompt, createdAt, lastFiredAt }`. On REPL start, re-arms any schedules whose next-fire has passed since last open — catches up overdue runs exactly once, then resumes normal cadence.
- **New slash commands** in `brainrouter-cli/src/cli/commands/obs.ts`:
  - `/schedule add <cron-or-time> <prompt>` — register a new schedule.
  - `/schedule list` — show all registered schedules with next-fire times.
  - `/schedule cancel <id>` — remove a schedule.
- `/loop` stays for interval-only use; `/schedule` is the new cron/absolute-time entry point.

**Files.**

- `brainrouter-cli/src/runtime/loopRunner.ts`
- `brainrouter-cli/src/cli/commands/obs.ts`
- `brainrouter-cli/src/tests/` (new `schedule.test.ts`)

**Acceptance.**

- `parseSchedule` handles cron, ISO datetime, and natural-language shapes; returns `undefined` for garbage input.
- `/schedule add` persists a record; `/schedule list` shows it with correct next-fire time; `/schedule cancel` removes it.
- REPL restart fires a missed cron exactly once, then resumes cadence from the new next-fire.

**Size:** M (~300 LoC + tests).

## 8. Strict Tool-Call Recovery *(carried from 0.3.7, pattern from deer-flow)*

**Problem.** When OpenAI-compatible reasoning models receive malformed tool histories — forced-stop mid-batch, provider-injected metadata, dangling tool calls that never received a result — they choke and refuse to continue. The current recovery is "throw and let the user retry."

**Change.**

- **Strip provider metadata on forced-stop.** Before re-injecting chat history on the next turn, scrub provider-internal fields (`finish_reason: 'tool_calls'` without a follow-up, OpenAI's `refusal` blocks on partial completions) that reasoning models treat as authoritative even after the user aborted.
- **Inject placeholder tool results** for any `tool_call` in the history that lacks a corresponding `tool` message. Use the envelope:
  ```json
  { "ok": false, "error": "tool call aborted before completion; safe to ignore" }
  ```
  so the model sees a complete request/response cycle even after a `^C` mid-batch.
- **Detection in `ensureWellFormedHistory(messages)`** — called at the top of `runTurn` once and before any compaction pass.

**Files.**

- `brainrouter-cli/src/agent/agent.ts` — add `ensureWellFormedHistory`, call at turn start
- `brainrouter-cli/src/tests/agent-runtime.test.ts` — cases for dangling tool calls, provider metadata scrub, ^C mid-batch recovery

**Acceptance.**

- History with a dangling `tool_call` (no paired `tool` result) gets a synthetic result injected before the next model call.
- Provider `refusal` blocks on partial completions are stripped from re-injected history.
- Existing well-formed histories pass through unchanged with no performance regression.

**Size:** S–M (~200 LoC + targeted tests). **Reference:** [`openSrc/deer-flow/backend/packages/harness/deerflow/runtime/`](../openSrc/deer-flow/backend/packages/harness/deerflow/runtime/) — "Strict Tool-Call Recovery" pattern.

## 10. Anthropic native `/v1/messages` adapter *(deferred from 0.3.7)*

**Problem.** The chat loop drives `/v1/chat/completions` for all providers including Anthropic. Claude users currently go through the OpenRouter OpenAI-compat gateway — it works identically but means we can't use Anthropic-native features: `tool_use` blocks, `thinking: { budget_tokens }` extended thinking, citations, or the Files API.

**Change.**

- Add a provider-dispatch layer in `agent.ts` (or a new `runtime/llmClient.ts`) that checks `config.llm.provider === 'anthropic'` and routes to a native `POST /v1/messages` call instead of `/v1/chat/completions`.
- Message conversion: translate the OpenAI message array to Anthropic's `{ role, content }` shape. Map `tool_calls` → `tool_use` blocks; `tool` results → `tool_result` content blocks.
- Tool schema conversion: translate OpenAI `tools[].function` → Anthropic `tools[]` (`name`, `description`, `input_schema`).
- Streaming: use Anthropic's SSE event stream (`content_block_delta`, `message_delta`) and map to the same `onToken`/`onToolStart`/`onToolEnd` callbacks the Ink UI already consumes.
- Expose `thinking: { type: "enabled", budget_tokens: N }` when `config.llm.effort === 'high'` and model supports it (`claude-opus-*`, `claude-sonnet-*` ≥ 3.7).
- Keep the OpenRouter gateway path as the fallback for users who prefer it (`config.llm.useGateway: true`).

**Files.**

- `brainrouter-cli/src/agent/agent.ts` or new `brainrouter-cli/src/runtime/llmClient.ts`
- `brainrouter-cli/src/config/config.ts` (add `useGateway` flag)
- `brainrouter-cli/src/tests/agent-runtime.test.ts`

**Acceptance.**

- `brainrouter` with `provider: anthropic` and a direct Anthropic API key sends requests to `api.anthropic.com/v1/messages`, not OpenRouter.
- Tool calls round-trip correctly through the `tool_use` / `tool_result` block format.
- Extended thinking activates on `effort: high` with a supported Claude model and the thinking tokens appear in scrollback.
- OpenRouter gateway path still works when `useGateway: true`.

**Size:** M–L (~500 LoC + tests).

## 11. `@mentions` tab completion *(deferred from 0.3.7)*

**Problem.** Slash discovery uses the inline Ink palette. `@<query>` file-picker is not yet wired — users have to type full paths by hand when referencing files in a prompt.

**Change.**

- In the Ink chat composer, detect when the buffer contains `@` followed by a non-space sequence.
- On `@<query>`, open an inline file-picker panel (below the composer, same overlay slot used by the slash palette) filtered by `query` against workspace files via `glob` + fuzzy ranking.
- On selection, replace `@<query>` with the resolved relative path.
- Support multi-`@` in one message (each `@` triggers its own picker in sequence).
- `@` with no query shows recently edited files (last 10 from `git diff --name-only HEAD` or workspace file scan if no git).

**Files.**

- `brainrouter-cli/src/cli/ink/ChatApp.tsx` — composer input handler
- `brainrouter-cli/src/cli/ink/SlashPalette.tsx` — extend or fork for `@` picker panel
- `brainrouter-cli/src/cli/ink/runChat.tsx` — wire file scan

**Acceptance.**

- Typing `@src/` in the composer opens a file picker filtered to `src/`.
- Selecting a file replaces `@src/` with the chosen path in the composer buffer.
- Pressing Escape dismisses the picker and leaves the raw `@...` text for manual editing.
- Multiple `@` mentions in one message all resolve before submit.

**Size:** M (~300 LoC + tests).

## 12. Prefix-aware memory tool lookup in briefing + `/mcp` doctor

**Problem.** After the 0.3.7 identity-based prefix work, the MCP pool exposes BrainRouter tools as `mcp_brainrouter_memory_recall` (or `mcp__brainrouter__memory_recall` with double-underscore identity). Several CLI sites still do raw `toolNames.has('memory_recall')` lookups against `listTools()` output, so the bare-name check always misses and the gated code silently skips. Symptom: chat shows `🧠 Briefing: 0 records from (none)` even when the brain is connected and `/mcp` shows `mcp_brainrouter_memory_recall` is exposed. **Single root cause** — this is not a server-side / missing-tool problem.

**Naming convention decision (binding for this fix and all future work).** Standardise on the **single-underscore** form `mcp_<server>_<tool>` everywhere — emission, prefix-stripping, lookup helpers, docs, and tests. The double-underscore `mcp__<server>__<tool>` form is **deprecated**. As part of this task, audit every site that currently accepts both shapes (e.g. [`isBrainOnline`](../brainrouter-cli/src/prompt/systemPrompt.ts), [`stripMcpPrefix`](../brainrouter-cli/src/cli/ink/toolFormat.ts)) and collapse them to the single-underscore form. Servers that still emit double-underscore names should be normalised at the pool boundary in [`mcpPool.ts`](../brainrouter-cli/src/runtime/mcpPool.ts) before tools are surfaced to the rest of the CLI.

**Root cause files.**

- [`brainrouter-cli/src/memory/briefing.ts`](../brainrouter-cli/src/memory/briefing.ts) lines 57, 60, 63 — primary cause. Gates `memory_recall`, `memory_working_context`, `memory_task_state` task queueing. With prefixed names the `Set.has` returns false → zero tasks → empty `sourcesQueried` → renders as `(none)`.
- [`brainrouter-cli/src/cli/commands/ui.ts`](../brainrouter-cli/src/cli/commands/ui.ts) lines 124-128 — `/mcp` doctor reports `memory_recall: not exposed` for the same reason, misleading users into thinking the server is broken.

**Not affected (but document the contract).** `callMcpTool(client, 'memory_recall', ...)` call sites in [`memory.ts`](../brainrouter-cli/src/cli/commands/memory.ts), [`consolidation.ts`](../brainrouter-cli/src/memory/consolidation.ts), and `briefing.ts` itself work because [`McpClientPool.callTool`](../brainrouter-cli/src/runtime/mcpPool.ts) resolves the prefix internally. The bug is exclusively in the `has()`-style discovery checks.

**Change.**

- Normalise tool names at the pool boundary ([`mcpPool.ts`](../brainrouter-cli/src/runtime/mcpPool.ts)): collapse any `mcp__<server>__<tool>` emission to single-underscore `mcp_<server>_<tool>` before they reach the rest of the CLI.
- Add a small helper (e.g. `hasMcpTool(toolNames, 'memory_recall')`) in [`runtime/mcpUtils.ts`](../brainrouter-cli/src/runtime/mcpUtils.ts) that matches `mcp_<server>_<tool>` (single-underscore only, post-normalisation) alongside the bare name.
- Replace the three `toolNames.has('memory_*')` calls in `briefing.ts` and the loop body in `ui.ts` with the helper.
- Collapse the dual-shape regexes in [`stripMcpPrefix`](../brainrouter-cli/src/cli/ink/toolFormat.ts) and [`isBrainOnline`](../brainrouter-cli/src/prompt/systemPrompt.ts) to single-underscore-only.
- Regression test in [`brainrouter-cli/src/tests/prompt.test.ts`](../brainrouter-cli/src/tests/prompt.test.ts) or a new `briefing.test.ts`: given a pool that lists `mcp_brainrouter_memory_recall`, `buildMemoryBriefing` must enqueue the recall task and `sourcesQueried` must contain `memory_recall`.

**Acceptance.**

- Chat with a connected brainrouter MCP profile shows `🧠 Briefing: N records from memory_recall, …` (never `(none)` when memory tools are exposed).
- `/mcp` doctor lists `memory_recall: available` when the tool is connected with a prefix.
- `callMcpTool` behavior unchanged.
- No remaining references to the `mcp__<server>__<tool>` double-underscore shape anywhere outside the pool's normalisation layer — confirmed via grep in CI.

**Files.**

- `brainrouter-cli/src/runtime/mcpPool.ts` (normalisation at boundary)
- `brainrouter-cli/src/runtime/mcpUtils.ts` (new `hasMcpTool` helper)
- `brainrouter-cli/src/memory/briefing.ts`
- `brainrouter-cli/src/cli/commands/ui.ts`
- `brainrouter-cli/src/prompt/systemPrompt.ts` (collapse dual-shape regex in `isBrainOnline`)
- `brainrouter-cli/src/cli/ink/toolFormat.ts` (collapse dual-shape regex in `stripMcpPrefix`)
- `brainrouter-cli/src/tests/briefing.test.ts` (new) or extend existing test

**Size:** S (~60 LoC + test, was XS before the standardisation scope). **Target:** 0.3.8.
