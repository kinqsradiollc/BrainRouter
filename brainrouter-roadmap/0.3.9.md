# 0.3.9 Roadmap — Memory Briefing + Cache-First Loop

**Status.** Items 1–7 merged into `feature/0.3.9-memory-briefing` and the
`release/0.3.9` branch. Items 8–14 (Reasonix-inspired cache-first /
repair / cost-control) are designed and not yet implemented in this
release. This remains the pre-0.4.0 release — no federation, no
cross-CLI inboxes, no typed cross-vendor delegation.

**Theme.** Two threads land together:

1. **Memory quality** (items 1–7, shipped) — better timing, better
   source selection, better compression, and better evidence when the
   agent decides what to brief.
2. **Cache-first loop + repair + cost-control** (items 8–14, new) —
   adapt the DeepSeek-Reasonix architecture to a provider-neutral CLI
   so prefix caching survives long sessions, tool-call failures
   self-repair, and per-turn USD cost becomes visible.

The thesis: BrainRouter is unique because the *brain* (MCP memory
recall) pins into the *byte-stable prefix*. Reasonix proved
99.82% cache hit (single user, 2026-05-01, ~$1.38 vs ~$61 uncached on
the same workload — see
`openSrc/DeepSeek-Reasonix/benchmarks/real-world-cache/README.md`) by
freezing `system + tool_specs` for the session. Our angle: freeze
`system + tool_specs + memory anchor cards` so memory continuity
and prefix cache reinforce each other instead of fighting.

## Why 0.3.9 Now Spans Both Threads

0.3.8 fixed delegation reliability and the prefixed-tool briefing bug.
0.3.9 first closed the memory-quality gap (items 1–7). The remaining
product gap is **cost shape and reliability** — and the Reasonix
reference made it concrete enough to land before 0.4.0 federation:

- Every turn currently rewrites parts of the system prompt (model
  family overlay, briefing block, mode/policy overlay), which breaks
  the provider prefix cache. We pay miss-rate on bytes that could be
  hit-rate.
- The briefing block was redesigned in item 1 to fire smartly, but the
  *bytes it injects* still live in the system message — they should
  live in a pinned, append-only synthetic message at the prefix tail so
  they stay byte-identical across turns.
- We have `toolCallRecovery.ts` (dedup + orphan synthesis + stalled
  preamble), but no scavenge / storm / truncation / schema-flatten
  passes. Small / OS models silently waste turns when their tool calls
  leak into `reasoning_content`, repeat themselves, or truncate mid-JSON.
- We expose `/tokens` and aggregate transcript stats, but no per-turn
  USD cost, no `prompt_cache_hit` accounting, no cache savings vs
  no-cache baseline.

Items 8–14 close those gaps without touching federation or the brain
job queue.

## 8. Cache-First Context Regions

**Problem.** Every turn currently rewrites large pieces of the system
prompt (briefing block, mode/policy overlay, model-family overlay,
goal anchor). DeepSeek's prefix cache, OpenAI's automatic cache, and
Anthropic's explicit `cache_control` all key on byte equality of the
prefix — any rewrite resets the hit rate to ~0. We pay miss-rate on
bytes that should be hit-rate.

**Change.**

- Add a `ContextRegions` module that partitions session state into
  three regions, mirroring
  `openSrc/DeepSeek-Reasonix/src/memory/runtime.ts`:
  - `ImmutablePrefix` — system prompt + tool specs + few-shots +
    pinned memory anchor cards. Computed once per session, hashed,
    pinned. Re-pinned only on `/refresh`, model change, MCP tool
    inventory change, or active-skill latch change.
  - `AppendOnlyLog` — turn messages, monotonically appended.
    `compactInPlace()` is the only break path, reserved for
    `/compact` and recovery.
  - `VolatileScratch` — reasoning content, plan state, per-turn
    notes. Distilled before any information is folded into the log;
    never sent upstream verbatim.
- Expose `prefix.fingerprint` (SHA-256 of system + tools + anchors,
  truncated to 16 hex chars).
- Cache-bust events log a `prefixChurn` warning with the diff so we
  can audit accidental rewrites.

**Files.**

- `brainrouter-cli/src/runtime/contextRegions.ts` *(new)*.
- `brainrouter-cli/src/runtime/prefixFingerprint.ts` *(new)*.
- [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts) — consume regions.
- [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts) — emit a base prefix, not per-turn rewrites.
- `brainrouter-cli/src/tests/contextRegions.test.ts` *(new)*.

**Acceptance.**

- `prefix.fingerprint` is stable across turns within a session for
  the same `(system, tools, anchors, model)` tuple.
- Verbose-mode logging shows `prefixChurn` when the fingerprint
  changes, with which field changed.
- `/refresh` invalidates the prefix and emits one cache-miss turn.

**Size:** M.

**Reference.** `openSrc/DeepSeek-Reasonix/src/memory/runtime.ts`
(`ImmutablePrefix`, `AppendOnlyLog`, `VolatileScratch`, fingerprint
verification with explicit drift error).

## 9. Prefix-Pinned Memory Briefing *(BrainRouter unique)*

**Problem.** Memory briefing bytes currently live inside the system
message, which the prompt builder rewrites every turn. Even when the
*content* is identical, byte-level shifts in surrounding overlays
push the briefing out of the cached prefix. Worse: when the briefing
content changes mid-session (a fresh recall card appears), the whole
prefix invalidates.

**Change.**

- Move the briefing block out of the system message and into a
  *pinned synthetic message pair* at the prefix tail:
  - `assistant` (or `system`-role per provider): "I have these
    memory cards pinned for this session: …" followed by the
    compacted card list.
  - The pair lives inside `ImmutablePrefix` (item 8), not in
    `AppendOnlyLog`.
- The pinned cards are produced by:
  1. Running the first-turn briefing decision from item 1.
  2. Pushing the result through TokenJuice-lite compaction (item 3)
     so the pinned bytes are minimal.
  3. Recording the anchor record IDs in the fingerprint payload so
     the prefix invalidates *only* when the anchors change, not on
     every recall.
- Mid-session full re-briefings *append* to `AppendOnlyLog` as a new
  synthetic tool result; they do not rewrite the pinned bytes. The
  pinned bytes are only refreshed by `/refresh-memory` or when the
  user closes / reopens the session.
- Cross-session reuse: when the user runs `brainrouter chat
  --session <key>`, the pinned bytes are re-derived from the same
  anchor IDs (the brain MCP must return byte-stable card payloads
  for the same anchor IDs, which is already the case for the
  recall-fingerprint hash path on the brain side).
- Add `BRAINROUTER_PREFIX_MEMORY_ANCHORS=on|off` env var, default
  `on`. Off falls back to the current in-system-message behavior.

**Files.**

- `brainrouter-cli/src/memory/anchorPin.ts` *(new)*.
- [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts) — call `anchorPin` once at session start, route mid-session briefings to `AppendOnlyLog`.
- [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts) — drop the briefing block from the system message; leave a one-line "Pinned memory below this message" marker.
- [`brainrouter-cli/src/memory/briefing.ts`](../brainrouter-cli/src/memory/briefing.ts) — emit the new synthetic message shape.
- [`brainrouter-cli/src/cli/commands/memory.ts`](../brainrouter-cli/src/cli/commands/memory.ts) — add `/refresh-memory`.
- `brainrouter-cli/src/tests/anchorPin.test.ts` *(new)*.

**Acceptance.**

- First-turn briefing pins to the prefix; second turn shows the same
  pinned bytes (byte-identical, verified by fingerprint).
- Adding a recall card mid-session does not invalidate the prefix.
- `/refresh-memory` does invalidate the prefix and re-pins.
- `BRAINROUTER_PREFIX_MEMORY_ANCHORS=off` returns to the current
  behavior with no regression.

**Size:** M.

**Reference.** No direct counterpart in Reasonix — they have
`REASONIX.md` as a project-memory file loaded into the system
prompt, but no MCP-mediated cross-session memory. This item is
where BrainRouter's brain plugs into Reasonix's cache-first loop and
becomes a real differentiator.

## 11. Tool-Call Repair Pipeline

**Problem.** [`toolCallRecovery.ts`](../brainrouter-cli/src/agent/toolCallRecovery.ts) currently
handles three failure modes: duplicate tool_call ids, orphan tool
calls (no matching result), and stalled preambles. Four more fail
silently:

- **Flatten miss** — small / OS models drop arguments on schemas
  with >10 leaves or depth >2 (the MCP `memory_*` family, our
  workflow tools, the agent registry payloads).
- **Scavenge miss** — models emit a tool call inside
  `reasoning_content` (or DSML markup in regular content) and forget
  to populate `tool_calls`. The call is lost; the next turn re-asks
  for the same work.
- **Truncation miss** — `max_tokens` cuts off a JSON argument
  mid-structure. We return a parse error; the model retries from
  scratch instead of resuming.
- **Storm miss** — the model calls the same `(name, args)` tuple
  three times in a row; we let all three through, burning tokens
  and rate-limit headroom.

**Change.**

- Reorganise `brainrouter-cli/src/agent/` into a small `repair/`
  directory:
  - `flatten.ts` — analyse each registered tool's JSON schema; if
    leaves >10 OR depth >2, present a flattened dot-notation schema
    to the model; re-nest at dispatch time. (Adapted from
    `openSrc/DeepSeek-Reasonix/src/repair/flatten.ts`.)
  - `scavenge.ts` — sweep both `reasoning_content` and `content`
    for JSON / DSML tool calls the model "forgot" to declare; coerce
    + emit if `(name, args)` is novel. (Adapted from
    `openSrc/DeepSeek-Reasonix/src/repair/scavenge.ts`.)
  - `truncation.ts` — detect unbalanced JSON in tool arguments;
    repair by closing braces or requesting continuation; surface a
    `TRUNCATION UNRECOVERABLE` note when impossible. (Adapted from
    `openSrc/DeepSeek-Reasonix/src/repair/truncation.ts`.)
  - `storm.ts` — sliding window of recent `(name, args)`; suppress
    when count ≥ threshold; mutating calls clear prior read-only
    entries (so a post-edit re-read isn't flagged). (Adapted from
    `openSrc/DeepSeek-Reasonix/src/repair/storm.ts`.)
  - `index.ts` — composes the four passes; exports
    `ToolCallRepair.process(declaredCalls, reasoning, content)`.
- Migrate the existing `dedupeToolCalls` /
  `synthesizeOrphanResults` / `parseArgumentsOrError` /
  `looksLikeStalledPreamble` /  `suggestSimilarToolName` from
  [`toolCallRecovery.ts`](../brainrouter-cli/src/agent/toolCallRecovery.ts) into the new directory or leave as separate
  utilities — they're complementary, not redundant.
- Wire `ToolCallRepair.process()` into `Agent.runTurn()` between
  the LLM response and tool dispatch.
- Per-turn repair report (`scavenged`, `truncationsFixed`,
  `stormsBroken`, `flattenedTools`) surfaces in verbose tracing and
  `/briefing` debug.

**Files.**

- `brainrouter-cli/src/agent/repair/{flatten,scavenge,truncation,storm,index}.ts` *(new directory)*.
- [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts) — call `ToolCallRepair.process()` per turn.
- [`brainrouter-cli/src/orchestration/tools.ts`](../brainrouter-cli/src/orchestration/tools.ts) — apply `flattenSchema` at registration when `analyzeSchema().shouldFlatten` is true.
- `brainrouter-cli/src/tests/repair-{flatten,scavenge,truncation,storm}.test.ts` *(new)*.

**Acceptance.**

- A tool with 15 leaves auto-flattens and round-trips through
  `nestArguments` to the original nested shape.
- A reasoning-content-only tool call is scavenged and dispatched.
- A truncated JSON argument is rebalanced and dispatched.
- Three identical calls in a row reduce to one + one suppressed +
  one suppressed; repair report shows `stormsBroken=2`.
- Repair counts visible in `/briefing` debug.

**Size:** M-L.

**Reference.** `openSrc/DeepSeek-Reasonix/src/repair/{index.ts,
flatten.ts, scavenge.ts, truncation.ts, storm.ts}`.

## 12. Turn-End Tool-Result Auto-Shrink (Prefix-Preserving)

**Problem.** [`toolCompaction.ts`](../brainrouter-cli/src/prompt/toolCompaction.ts) (item 3) shrinks
*incoming* tool results before they enter the model context. That
helps on the way in, but does nothing for results that were
oversized on a *prior* turn and now drag through every cached
prefix request. The full result was useful when read; it isn't
useful (at full size) ten turns later.

**Change.**

- After each turn completes (no in-flight `tool_calls`), walk the
  `AppendOnlyLog` and replace any `role: tool` content whose token
  count exceeds `TURN_END_RESULT_CAP_TOKENS` (default 3000) with the
  same TokenJuice-lite compaction output as item 3.
- Critical invariant: this *appends* a compacted replacement to the
  log only via the controlled `compactInPlace()` path (item 8);
  the prefix remains stable because the prefix never contained
  tool results in the first place. The cache miss is bounded to
  the affected tool-result message — older turns stay cached, and
  the compact log is byte-stable for subsequent turns.
- A proactive 40% context-ratio threshold runs the same shrink
  mid-iteration before the 80% emergency threshold; this prevents a
  multi-tool turn from blowing the context window before we can
  trim.
- Full raw outputs remain in [`brainrouter-cli/src/transcript/`](../brainrouter-cli/src/transcript/) (transcript layer)
  unchanged.
- Add `BRAINROUTER_TURN_END_RESULT_CAP_TOKENS` env override and a
  one-line `/tokens` row: `shrunk: 4 results (saved 18,200 tok)`.

**Files.**

- `brainrouter-cli/src/agent/turnEndShrink.ts` *(new)*.
- [`brainrouter-cli/src/prompt/toolCompaction.ts`](../brainrouter-cli/src/prompt/toolCompaction.ts) — export
  reusable per-message compactor.
- [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts) — call `runTurnEndShrink()` at
  turn boundary.
- `brainrouter-cli/src/tests/turnEndShrink.test.ts` *(new)*.

**Acceptance.**

- A 12KB tool result shrinks to ≤3000 tokens at the next turn end.
- The transcript still contains the full 12KB output.
- Cache hit % (item 10) does not regress after the shrink — the
  affected tool message bytes change once, then re-cache.

**Size:** M.

## 13. Model-Tier Self-Escalation

**Problem.** `/effort low|medium|high` (0.3.6) tunes reasoning depth
but does not switch model tiers. Users on multi-tier providers
(OpenAI gpt-5-mini → gpt-5 →
gpt-5-pro, DeepSeek v4-flash → v4-pro, OpenRouter
`gemini-2.5-flash` → `gemini-2.5-pro`) cannot let the *model* decide
when to escalate. The user either pre-arms `/model pro` and pays
12× across the whole session, or stays on flash and watches the
model loop on tasks it isn't strong enough to finish.

**Change.**

- Add a `tierLadder` table per provider:
  - `deepseek`: `v4-flash → v4-pro`
  - *(Anthropic native ladder removed in 0.3.9 — see Thread C.)*
  - `openai`: `gpt-5-mini → gpt-5 → gpt-5-pro`
  - `openrouter` / generic: configurable in
    `~/.config/brainrouter/config.json` under `tierLadder`.
- When the model emits `<<<NEEDS_HIGH>>>` (bare) or
  `<<<NEEDS_HIGH: <reason>>>>` as the first line of its response,
  the runtime aborts the current call and retries the same turn on
  the next-higher tier.
- The current tier (pro / top) is a no-op — pro is the top, so the
  contract tells the model it can't escalate further.
- Auxiliary calls (`runCompaction`, child-agent spawns, repair
  retries) always pin to the flash / mini tier regardless of user
  setting. There's no reason to pay pro rates for "paraphrase these
  tool results into prose".
- Surface the escalation in a yellow warning row in the Ink REPL
  (`⚠️ escalated to gpt-5 — reason: …`). Silent
  escalation is rejected.
- Add `/tier <flash|standard|pro>` slash command to pin a tier for
  the rest of the session.
- The system prompt's tier contract is added once at session start
  via the immutable prefix; no per-turn rewrite.

**Files.**

- `brainrouter-cli/src/runtime/tierLadder.ts` *(new)*.
- `brainrouter-cli/src/cli/commands/tier.ts` *(new)*.
- [`brainrouter-cli/src/agent/agent.ts`](../brainrouter-cli/src/agent/agent.ts) — detect `<<<NEEDS_HIGH>>>`, abort, retry.
- [`brainrouter-cli/src/prompt/systemPrompt.ts`](../brainrouter-cli/src/prompt/systemPrompt.ts) — add a small tier-contract block to the prefix.
- `brainrouter-cli/src/tests/tier.test.ts` *(new)*.

**Acceptance.**

- `<<<NEEDS_HIGH>>>` triggers exactly one retry on the next tier.
- `<<<NEEDS_HIGH>>>` from a pro-tier turn is a no-op.
- Auxiliary calls (compaction, repair retry) never escalate.
- The escalation reason appears in the Ink warning row.
- `/tier pro` pins the rest of the session regardless of marker.

**Size:** M.
