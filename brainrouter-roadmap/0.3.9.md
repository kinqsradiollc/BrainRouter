# 0.3.9 Roadmap — Memory Briefing, Source Awareness, and CLI Context Parity

**Status.** Planned. This is the pre-0.4.0 release. Do not start
federation, cross-CLI inboxes, or typed cross-vendor delegation here.

**Theme.** Make BrainRouter's own CLI and MCP memory path behave like a
serious local coding memory layer before the 0.4.0 federation jump. The
focus is not "more memories"; it is better timing, better source
selection, better compression, and better evidence when the agent decides
what to brief.

## Why 0.3.9 Exists

0.3.8 fixed delegation reliability and the prefixed-tool briefing bug.
The remaining product gap is memory use quality:

- The CLI resolves a session at startup, but memory briefing is not only a
  startup feature. `Agent.runTurn()` calls `injectRecallContext()` before
  turns, and `buildMemoryBriefing()` can query recall, working memory, and
  task state.
- The current auto-briefing is gated: first turn, post-compaction, or
  prompts with at least two entity-shaped tokens. Thin turns get only a
  memory hint and rely on the model to manually call recall.
- OpenHuman's strongest transferable pattern is a source-aware memory
  stack: ingestion jobs, source chunks, tree summaries, drill-down/fetch
  tools, vault mirror, event-bus observability, and TokenJuice-style tool
  output compaction before context injection.
- Claude Code's strongest transferable pattern in the vendored material is
  not hidden memory logic; it is stable workflow surface: plugins,
  commands, agents, hooks, managed settings, and guardrail hooks that
  shape the agent before and around tool use.

0.3.9 should make BrainRouter's briefing smarter and inspectable without
moving the whole brain-agent/source-tree roadmap forward too early.

## Release Goals

1. **Brief at the right moments.** Replace the blunt entity-token gate with
   a reasoned trigger engine that can tell when memory is useful during a
   session, not just at session start.
2. **Brief the right sources.** Pull from recall, working memory, task
   state, file history, failed attempts, and recall explanations based on
   request shape.
3. **Pack context like a product.** Apply TokenJuice-style compaction to
   tool outputs and memory snippets before they enter the model context.
4. **Make memory decisions inspectable.** `/briefing` should explain why a
   briefing fired or skipped, what sources were queried, what was omitted,
   and what token budget was saved.
5. **Keep 0.4.0 clean.** Do not implement federation, shared active
   session registry, cross-CLI messaging, brain-agent job queue, source
   chunk tables, full tree storage, or vault mirror in this release.

## 1. Adaptive Memory Briefing Trigger Engine

**Problem.** Current gated recall is cheap but too blunt. It fires on
first turn, post-compaction, or `countEntityTokens(prompt) >= 2`, and
otherwise inserts a lightweight hint. That saves tokens, but it can miss
important memory moments: "continue", "same issue", "that file again",
"fix the regression", "what did we decide", short prompts after tool
bursts, and post-error pivots.

**Change.**

- Replace the boolean gate inside `Agent.injectRecallContext()` with a
  small trigger engine.
- Emit a structured decision:
  `fire | skip | hint-only`, `reasons[]`, `query`, `sourcePlan`, and
  `budget`.
- Trigger on:
  - first turn and post-compaction;
  - explicit memory/previous/history/continue/handoff wording;
  - file path or symbol mentions;
  - active `/goal` with new iteration or blocker state;
  - recent failed tool call or loop-limit recovery;
  - child-agent result synthesis;
  - enough elapsed turns since the last full briefing;
  - user prompt shorter than a threshold but conversation state implies
    continuation.
- Keep `BRAINROUTER_RECALL_MODE=always|gated|off`, but make `gated`
  use the new decision engine.

**Files.**

- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter-cli/src/memory/briefing.ts`
- `brainrouter-cli/src/tests/memory.test.ts`
- `brainrouter-cli/src/tests/agent-runtime.test.ts`

**Acceptance.**

- "continue", "same issue", and file-specific prompts trigger full
  briefing after the first turn.
- "thanks" and purely social short replies skip full briefing.
- Post-compaction always forces one full briefing.
- Skip decisions are visible through `/briefing`.

**Size:** M.

## 2. Source-Aware Briefing Router

**Problem.** `buildMemoryBriefing()` currently queues a fixed set of
sources when available: `memory_recall`, `memory_working_context`, and
sometimes `memory_task_state`. The system prompt tells the model to call
`memory_file_history`, `memory_failed_attempts`, and
`memory_explain_recall`, but the auto-briefing path does not yet route
between sources with intent.

**Change.**

- Add a `BriefingSourcePlan` that can select:
  - `memory_recall` for general cognitive context;
  - `memory_working_context` for long-running task continuity;
  - `memory_task_state` only when no active goal anchor owns current
    objective state;
  - `memory_explain_recall` when the result set is thin, surprising, or
    high-risk;
  - `memory_file_history` when prompts include paths/symbols;
  - `memory_failed_attempts` when prompt or recent transcript indicates a
    retry/debugging loop.
- Make every source return a normalized card shape with:
  `source`, `recordId?`, `type?`, `confidence?`, `reason`, `preview`,
  `tokenEstimate`.
- Preserve existing behavior when optional MCP tools are absent.

**Files.**

- `brainrouter-cli/src/memory/briefing.ts`
- `brainrouter-cli/src/memory/formatters.ts`
- `brainrouter-cli/src/runtime/mcpUtils.ts`
- `brainrouter-cli/src/tests/memory.test.ts`
- `brainrouter/src/tools/memory-explain.ts`
- `brainrouter/src/tools/memory-engineering.ts`

**Acceptance.**

- File-specific prompts include file-history cards when the tool exists.
- Debug/retry prompts include failed-attempt or debug-trace cards when the
  tool exists.
- Missing optional tools degrade silently to the current recall/working
  memory path.
- The briefing never dumps raw 70KB JSON into the prompt.

**Size:** M.

## 3. TokenJuice-Lite CLI Context Compaction

**Problem.** BrainRouter clamps large tool results at a fixed character
limit, and working memory can offload child outputs, but tool outputs are
not semantically reduced before entering the model-visible trace. That is
weaker than OpenHuman's TokenJuice pattern, where command/tool outputs
are classified and compacted with command-specific rules.

**Change.**

- Add a small TypeScript compaction package for the CLI context path:
  - classify by tool/command (`git status`, `npm test`, `tsc`, `rg`,
    `docker`, generic JSON, generic text);
  - keep errors, filenames, counts, and summaries;
  - drop repeated progress lines, ANSI noise, and long unchanged chunks;
  - return `inlineText`, `omittedChars`, `ruleId`, `confidence`.
- Apply before LLM-visible tool result insertion in `Agent.runTurn()`.
- Keep full raw output in transcript exactly as today.
- Add a kill switch: `BRAINROUTER_CONTEXT_COMPACTION=false`.

**Files.**

- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter-cli/src/prompt/compactor.ts` or new
  `brainrouter-cli/src/prompt/toolCompaction.ts`
- `brainrouter-cli/src/tests/tool-format.test.ts`
- `brainrouter-cli/src/tests/agent-runtime.test.ts`

**Acceptance.**

- Failing `npm test`, `tsc`, and `rg` outputs keep actionable failures and
  paths.
- Long successful outputs shrink substantially before model insertion.
- Transcript still stores the full uncompressed result.
- `/tokens` can include compacted chars avoided.

**Size:** M.

## 4. Briefing Inspector

**Problem.** The user can see that a briefing happened, but not enough
about why it happened or why it skipped. This makes it hard to debug
"the agent forgot" versus "the gate skipped" versus "recall returned
empty".

**Change.**

- Extend `Agent.getLastBriefing()` with:
  `decision`, `reasons`, `sourcesPlanned`, `sourcesQueried`,
  `recordIds`, `recordCount`, `tokensInjected`, `tokensSaved`,
  `skippedSources`, `warnings`.
- Update `/briefing` to render:
  - latest decision;
  - source table;
  - recalled record IDs;
  - compaction stats;
  - suggested manual recall command when skipped.
- Surface this in quiet mode only when explicitly requested.

**Files.**

- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter-cli/src/cli/commands/memory.ts`
- `brainrouter-cli/src/cli/ink/runChat.tsx`
- `brainrouter-cli/src/tests/memory.test.ts`

**Acceptance.**

- `/briefing` tells the user whether full briefing fired, skipped, or
  hint-only fired.
- It explains skip reasons without exposing secrets or raw memory payloads.
- It shows source-level token/cost estimates.

**Size:** S.

## 5. Read-Only Source Sync Spike

**Problem.** OpenHuman's source ingestion/tree/vault stack is too large
for 0.3.9, but BrainRouter needs a pre-0.4 proof that source-aware memory
can work for local code/docs without polluting the durable schema too
early.

**Change.**

- Add a read-only spike command or hidden dev command that scans a small
  local source set and produces an ephemeral source manifest:
  `path`, `mtime`, `hash`, `size`, `kind`, `title`, `summary?`.
- Exclude `.git`, `node_modules`, `dist`, `.next`, build artifacts,
  benchmark datasets, and large files.
- Do not create source chunk tables.
- Do not create a vault mirror.
- Use the manifest only as an input to source-aware briefing experiments
  and benchmark cases.

**Files.**

- `brainrouter-cli/src/cli/commands/memory.ts` or new command module
- `brainrouter-cli/src/memory/sourceManifest.ts`
- `brainrouter-cli/src/tests/memory.test.ts`

**Acceptance.**

- The spike can scan the current repo and return a bounded manifest.
- It never writes into the project tree by default.
- It proves which file kinds and summaries would be useful for the 0.4.x
  source chunk/vault work.

**Size:** S-M.

## 6. Hook/Policy Memory Guardrails

**Problem.** Claude Code's vendored examples show a simple but useful
pattern: hooks and managed settings can shape behavior around tools
without relying on the model to remember every rule. BrainRouter already
has hook stores and Hookify, but memory-specific policies are not first
class.

**Change.**

- Add memory policy hooks for:
  - blocking memory capture of obvious secrets;
  - warning when recall returns stale/off-workspace records;
  - requiring source/evidence for high-confidence engineering memories;
  - preferring `rg` over broad recursive scans in generated source-sync
    prompts.
- Keep this as local CLI policy and MCP redaction integration, not an
  enterprise settings system.

**Files.**

- `brainrouter-cli/src/state/hookifyStore.ts`
- `brainrouter-cli/src/agent/agent.ts`
- `brainrouter/src/memory/redaction.ts`
- `brainrouter/src/memory/capture.ts`
- `brainrouter-cli/src/tests/runtime.test.ts`

**Acceptance.**

- Secret-looking content is blocked or redacted before capture.
- Stale/off-workspace memory warnings are visible in `/briefing`.
- Policy failures do not crash the user-facing turn.

**Size:** S-M.

## 7. Briefing Quality and Token-Savings Bench

**Problem.** We should not claim parity with OpenHuman-style memory
behavior by vibes. 0.3.9 needs a small benchmark harness that measures
whether better briefing triggers and compaction actually improve
continuity while saving tokens.

**Change.**

- Add scripted benchmark cases for:
  - first-turn recall;
  - short continuation prompt;
  - file-specific follow-up;
  - debugging retry after failure;
  - post-compaction recall;
  - child-agent synthesis after offload.
- Measure:
  - full briefing fired/skipped;
  - record hit count;
  - whether expected record IDs were surfaced;
  - injected tokens;
  - compacted chars avoided;
  - manual recall fallback suggested when skipped.
- Keep this separate from LongMemEval; it is a CLI briefing behavior
  benchmark, not a memory-quality research benchmark.

**Files.**

- `brainrouter-cli/src/tests/memory.test.ts`
- `brainrouter/benchmark/` or `brainrouter-cli/src/tests/briefing-bench.test.ts`
- `brainrouter-roadmap/0.3.9.md`

**Acceptance.**

- Benchmark can run locally without network by using stub MCP clients.
- It fails when expected trigger cases skip full briefing.
- It reports token/char savings from compaction.

**Size:** M.

## Out of Scope for 0.3.9

- 0.4.0 federation stages: `active_sessions`, `session_inbox`,
  cross-CLI messaging, handoff, and shared live session discovery.
- Synthesized typed `delegate_*` tools and cross-vendor task routing.
- Durable brain-agent job queue.
- Durable source chunk tables.
- Full memory tree bucket sealing.
- Obsidian-compatible vault mirror.
- Dashboard source tree browser.
- Composio-style OAuth/integration provider layer.

These stay in 0.4.0 / 0.4.x so 0.3.9 remains a shippable CLI memory
quality release.
